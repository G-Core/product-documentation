Build two FastEdge apps in Rust using the WASI-HTTP interface: a minimal hello-world component to verify the toolchain, and a realistic app that fetches live data from an external API and returns a shaped JSON response.

WASI-HTTP is the HTTP interface from the WASI 0.2 standard. This approach does not require the [Gcore Rust SDK](https://github.com/G-Core/FastEdge-sdk-rust) — the only runtime dependency is `wstd`, a lightweight async library that implements the standard WASI interfaces.

**Prerequisites:** [Rust and Cargo](https://rustup.rs). On Windows, also install [Visual Studio Build Tools](https://visualstudio.microsoft.com/downloads/#build-tools-for-visual-studio-2022) with the **Desktop development with C++** workload.

## Hello world

Before building something complex, it helps to verify the full toolchain works end-to-end: compile to WASI, upload, deploy, get a response. The hello-world component does one thing — it reads the request URL and echoes it back in the response body.

### Set up the project

1. Add the `wasm32-wasip2` compilation target. This is a one-time step:

   ```sh
   rustup target add wasm32-wasip2
   ```

2. Create a new library crate:

   ```sh
   cargo new --lib hello-world
   cd hello-world
   ```

3. Open `Cargo.toml` and replace its contents with:

   ```toml
   [package]
   name = "hello_world"
   version = "0.1.0"
   edition = "2021"

   [lib]
   crate-type = ["cdylib"]

   [dependencies]
   wstd = "0.6"
   anyhow = "1"
   ```

   The `[lib]` section is what changes. `crate-type = ["cdylib"]` tells Cargo to produce a C-compatible dynamic library — the format required by the WASI component model. Without it, Cargo builds a standard Rust library that cannot run as a WebAssembly component.

### Write the handler

Replace `src/lib.rs` with:

```rust
use wstd::http::body::Body;
use wstd::http::{Request, Response};

#[wstd::http_server]
async fn main(request: Request<Body>) -> anyhow::Result<Response<Body>> {
    let url = request.uri().to_string();

    Ok(Response::builder()
        .status(200)
        .header("content-type", "text/plain;charset=UTF-8")
        .body(Body::from(format!(
            "Hello, you made a wasi request to {url}"
        )))?)
}
```

The `#[wstd::http_server]` macro marks this function as the WASI-HTTP entry point — the runtime calls it for every incoming request. The function takes a `Request<Body>` and returns a `Response<Body>`, using the standard `http` crate types with `wstd`'s body implementation layered on top.

Note: `wstd` 0.6 requires the full generic form `Request<Body>`. Writing `Request` without the type argument causes a compiler error.

### Build

```sh
cargo build --release --target wasm32-wasip2
```

The first build downloads and compiles all dependencies — expect one to two minutes. The output binary is at `./target/wasm32-wasip2/release/hello_world.wasm`.

### Deploy and test

Deployment requires two API calls: upload the binary, then create the app referencing the returned binary ID. Set your [API token](/account-settings/api-tokens) before running the commands:

```sh
export GCORE_API_KEY="{YOUR_API_KEY}"
```

<Warning>
Do not commit API keys to source control.
</Warning>

**Upload the binary:**

```sh
curl -sX POST 'https://api.gcore.com/fastedge/v1/binaries/raw' \
  -H "Authorization: APIKey $GCORE_API_KEY" \
  -H 'Content-Type: application/octet-stream' \
  --data-binary "@./target/wasm32-wasip2/release/hello_world.wasm"
```

```json
{"id": 4695, "api_type": "wasi-http", "status": 1}
```

**Create the app** — replace `4695` with the `id` from the response above. Your binary ID and app URL will differ from the examples here:

```sh
export BINARY_ID=4695

curl -sX POST 'https://api.gcore.com/fastedge/v1/apps' \
  -H "Authorization: APIKey $GCORE_API_KEY" \
  -H 'Content-Type: application/json' \
  -d "{\"name\": \"my-hello-world\", \"binary\": $BINARY_ID, \"status\": 1}"
```

```json
{"name": "my-hello-world", "url": "https://my-hello-world-1000503.fastedge.app", "api_type": "wasi-http"}
```

**Test the app.** The URL becomes active within a few seconds:

```sh
curl -i https://my-hello-world-1000503.fastedge.app/test-path
```

```
HTTP/1.1 200 OK
content-type: text/plain;charset=UTF-8

Hello, you made a wasi request to http://my-hello-world-1000503.fastedge.app/test-path
```

The full toolchain works. Now build something useful with it.

## Fetch data from an external API

The hello-world component ignores the upstream world — it reads the request and returns a static response. The defining capability of a WASI-HTTP component is the ability to make *outbound* HTTP requests from inside the handler: call a live API, transform what comes back, and return the result to the caller.

This section builds a component that calls a REST API, takes the first five records from the response, and returns them as JSON.

<Info>
The examples use `jsonplaceholder.typicode.com`, a free placeholder API, as a stand-in for any REST endpoint. It is suitable for development and testing only — do not use it in production.
</Info>

### Set up the project

Create a separate project for this component:

```sh
cargo new --lib outbound-fetch
cd outbound-fetch
```

Replace `Cargo.toml`:

```toml
[package]
name = "outbound_fetch"
version = "0.1.0"
edition = "2021"

[lib]
crate-type = ["cdylib"]

[dependencies]
wstd = "0.6"
anyhow = "1"
serde_json = "1"
```

The only addition over hello-world is `serde_json` for parsing and building JSON. No derive macros are needed — this example treats the response as an untyped value tree, which is the right choice when slicing and reshaping existing JSON rather than mapping it to your own types.

### Write the handler

Replace `src/lib.rs`:

```rust
use anyhow::anyhow;
use wstd::http::body::Body;
use wstd::http::{Client, Request, Response};
use serde_json::{json, Value};

#[wstd::http_server]
async fn main(_request: Request<Body>) -> anyhow::Result<Response<Body>> {
    // Build the outbound request to the upstream API
    let upstream_req = Request::get("http://jsonplaceholder.typicode.com/users")
        .body(Body::empty())
        .map_err(|e| anyhow!("failed to build request: {e}"))?;

    // Send it — this goes through the WASI outbound-http interface,
    // not a native socket
    let client = Client::new();
    let upstream_resp = client
        .send(upstream_req)
        .await
        .map_err(|e| anyhow!("upstream request failed: {e}"))?;

    // Split the response and read the full body
    let (_, mut body) = upstream_resp.into_parts();
    let body_bytes = body.contents().await?;

    // Parse the JSON, take the first 5 users, and shape the response
    let users: Value = serde_json::from_slice(body_bytes)?;
    let sliced = match users.as_array() {
        Some(arr) => Value::Array(arr.iter().take(5).cloned().collect()),
        None => Value::Array(vec![]),
    };

    let result = json!({
        "users": sliced,
        "total": 5,
        "skip": 0,
        "limit": 30,
    });

    Ok(Response::builder()
        .status(200)
        .header("content-type", "application/json")
        .body(Body::from(result.to_string()))?)
}
```

Three patterns in this code are worth understanding before adapting it to a real endpoint:

**`Client::new()` and `client.send()`** use `wstd`'s implementation of the WASI outbound-http interface. The request goes through the host runtime — no native networking code lives inside the component. The `await` is real async: the handler yields while the upstream request is in flight.

**`upstream_resp.into_parts()`** splits the response into its status and headers (`Parts`) and an async body stream. `body.contents().await` reads the full stream into a byte slice before parsing. For a large upstream response this buffers everything in memory — if you only need to forward a stream rather than inspect it, the split lets you do that without buffering.

**Error propagation with `?`:** every fallible operation short-circuits back through `anyhow::Result`. FastEdge converts a handler that returns `Err` into a 500 response — the error message is available in application logs, not in the response body.

### Build

```sh
cargo build --release --target wasm32-wasip2
```

Output: `./target/wasm32-wasip2/release/outbound_fetch.wasm`.

### Deploy and test

Upload and deploy using the same pattern as hello-world:

```sh

  --data-binary "@./target/wasm32-wasip2/release/outbound_fetch.wasm"
```

```json

```


**On latency:** the outbound call happens on every request, from every edge node that handles traffic for this app. For data that changes infrequently, storing the upstream response in a [FastEdge KV store](/fastedge/kv-stores/manage-kv-store) eliminates that per-request latency after the first fetch.

## Clean up

To delete an app, use its `id` from the create-app response:

```sh
curl -sX DELETE "https://api.gcore.com/fastedge/v1/apps/{APP_ID}" \
  -H "Authorization: APIKey $GCORE_API_KEY"
```

The binary is not deleted when the app is deleted. To delete the binary as well:

```sh
curl -sX DELETE "https://api.gcore.com/fastedge/v1/binaries/{BINARY_ID}" \
  -H "Authorization: APIKey $GCORE_API_KEY"
```

A binary referenced by an active application cannot be deleted — remove the app first.
