FastEdge runs applications as WebAssembly components — portable binaries that execute on a global edge network without servers or containers. These components use [WASI-HTTP](https://github.com/WebAssembly/WASI) to receive requests and call external services, and `wstd` is the Rust library that implements it — the only dependency a FastEdge app in Rust needs.

A hello-world component verifies the full toolchain works end-to-end: compile, upload, deploy, get a response. A second, more realistic component calls an external REST API from inside the handler, transforms the response, and returns shaped JSON — demonstrating WASI-HTTP's defining capability.

[Rust and Cargo](https://rustup.rs) are required. On Windows, also install [Visual Studio Build Tools](https://visualstudio.microsoft.com/downloads/#build-tools-for-visual-studio-2022) with the **Desktop development with C++** workload.

## Hello world

The hello-world component does one thing — it reads the request URL and echoes it back in the response body.

### Project setup

`wasm32-wasip2` is the Rust compilation target for WASI 0.2 components — add it once to the toolchain and it is available for every project.

1. Add the `wasm32-wasip2` target:

   ```sh
   rustup target add wasm32-wasip2
   ```

2. Create a new library crate:

   ```sh
   cargo new --lib hello-world
   cd hello-world
   ```

3. Open `Cargo.toml` and replace its contents with:

   ```toml
   [package]
   name = "hello_world"
   version = "0.1.0"
   edition = "2021"

   [lib]
   crate-type = ["cdylib"]

   [dependencies]
   wstd = "0.6"
   anyhow = "1"
   ```

   The `[lib]` section is what changes — `crate-type = ["cdylib"]` tells Cargo to produce a C-compatible dynamic library, the format required by the WASI component model, rather than a standard Rust library.

### Handler

Replace `src/lib.rs` with:

```rust
use wstd::http::body::Body;
use wstd::http::{Request, Response};

#[wstd::http_server]
async fn main(request: Request<Body>) -> anyhow::Result<Response<Body>> {
    let url = request.uri().to_string();

    Ok(Response::builder()
        .status(200)
        .header("content-type", "text/plain;charset=UTF-8")
        .body(Body::from(format!(
            "Hello, you made a wasi request to {url}"
        )))?)
}
```

The `#[wstd::http_server]` macro marks this function as the WASI-HTTP entry point — the runtime calls it for every incoming request. It receives a `Request<Body>` and returns a `Response<Body>`, using the standard `http` crate types with `wstd`'s body implementation.

`wstd` 0.6 requires the full generic form `Request<Body>` — omitting the type argument causes a compiler error.

### Build

```sh
cargo build --release --target wasm32-wasip2
```

The first build downloads and compiles all dependencies — expect one to two minutes — and produces the binary at `./target/wasm32-wasip2/release/hello_world.wasm`.

### Deployment and testing

Deployment requires two API calls — upload the binary, then create the app using the returned binary ID. Both calls authenticate with a permanent [API token](/account-settings/api-tokens), so set it as an environment variable first:

```sh
export GCORE_API_KEY="{YOUR_API_KEY}"
```

<Warning>
Do not commit API keys to source control.
</Warning>

Upload the binary to get back a binary ID:

```sh
curl -sX POST 'https://api.gcore.com/fastedge/v1/binaries/raw' \
  -H "Authorization: APIKey $GCORE_API_KEY" \
  -H 'Content-Type: application/octet-stream' \
  --data-binary "@./target/wasm32-wasip2/release/hello_world.wasm"
```

```json
{"id": 4695, "api_type": "wasi-http", "status": 1}
```

Pass that `id` to create the app. The binary ID and app URL in the examples below will differ from the ones returned by the upload:

```sh
export BINARY_ID=4695

curl -sX POST 'https://api.gcore.com/fastedge/v1/apps' \
  -H "Authorization: APIKey $GCORE_API_KEY" \
  -H 'Content-Type: application/json' \
  -d "{\"name\": \"my-hello-world\", \"binary\": $BINARY_ID, \"status\": 1}"
```

```json
{"name": "my-hello-world", "url": "https://my-hello-world-1000503.fastedge.app", "api_type": "wasi-http"}
```

The URL becomes active within a few seconds — send a request to confirm the handler is running:

```sh
curl -i https://my-hello-world-1000503.fastedge.app/test-path
```

```
HTTP/1.1 200 OK
content-type: text/plain;charset=UTF-8

Hello, you made a wasi request to http://my-hello-world-1000503.fastedge.app/test-path
```

The full toolchain works. Now build something useful with it.

## Fetch data from an external API

While the hello-world component only returns a static response, WASI-HTTP's defining capability is making outbound HTTP requests from inside the handler — calling a live API, transforming the response, and returning the result to the caller. This component calls a REST API, takes the first five records from the response, and returns them as JSON.

<Info>
The examples use `jsonplaceholder.typicode.com`, a free placeholder API, as a stand-in for any REST endpoint. It is suitable for development and testing only — do not use it in production.
</Info>

### Project setup

Create a separate project for this component:

```sh
cargo new --lib outbound-fetch
cd outbound-fetch
```

The dependencies are almost identical to hello-world — the only addition is `serde_json` for parsing and building JSON. Replace `Cargo.toml`:

```toml
[package]
name = "outbound_fetch"
version = "0.1.0"
edition = "2021"

[lib]
crate-type = ["cdylib"]

[dependencies]
wstd = "0.6"
anyhow = "1"
serde_json = "1"
```

No derive macros are needed — this example treats the response as an untyped `Value` tree, which avoids defining structs just to slice and reshape an existing JSON payload.

### Handler

Replace `src/lib.rs`:

```rust
use anyhow::anyhow;
use wstd::http::body::Body;
use wstd::http::{Client, Request, Response};
use serde_json::{json, Value};

#[wstd::http_server]
async fn main(_request: Request<Body>) -> anyhow::Result<Response<Body>> {
    // Build the outbound request to the upstream API
    let upstream_req = Request::get("http://jsonplaceholder.typicode.com/users")
        .body(Body::empty())
        .map_err(|e| anyhow!("failed to build request: {e}"))?;

    // Send it — this goes through the WASI outbound-http interface,
    // not a native socket
    let client = Client::new();
    let upstream_resp = client
        .send(upstream_req)
        .await
        .map_err(|e| anyhow!("upstream request failed: {e}"))?;

    // Split the response and read the full body
    let (_, mut body) = upstream_resp.into_parts();
    let body_bytes = body.contents().await?;

    // Parse the JSON, take the first 5 users, and shape the response
    let users: Value = serde_json::from_slice(body_bytes)?;
    let sliced = match users.as_array() {
        Some(arr) => Value::Array(arr.iter().take(5).cloned().collect()),
        None => Value::Array(vec![]),
    };

    let result = json!({
        "users": sliced,
        "total": 5,
        "skip": 0,
        "limit": 30,
    });

    Ok(Response::builder()
        .status(200)
        .header("content-type", "application/json")
        .body(Body::from(result.to_string()))?)
}
```

The code follows three patterns that appear in most WASI-HTTP components.

`Client::new()` creates a WASI HTTP client that sends requests through the host runtime — not a native socket, so no system networking libraries are needed inside the component.

`upstream_resp.into_parts()` splits the response into its status/headers and an async body stream; `body.contents().await` then reads the full stream into memory before parsing. For responses that only need to be forwarded without inspection, the same split allows streaming the body without buffering it.

Every `?` propagates errors back through `anyhow::Result` — FastEdge converts a handler that returns `Err` into a 500 response, with the error message available in application logs rather than the response body.

### Build

```sh
cargo build --release --target wasm32-wasip2
```

The output binary is at `./target/wasm32-wasip2/release/outbound_fetch.wasm`.

### Deployment and testing

Upload and deploy using the same pattern as hello-world:

```sh
# Upload the binary
curl -sX POST 'https://api.gcore.com/fastedge/v1/binaries/raw' \
  -H "Authorization: APIKey $GCORE_API_KEY" \
  -H 'Content-Type: application/octet-stream' \
  --data-binary "@./target/wasm32-wasip2/release/outbound_fetch.wasm"
```

```json
{"id": 4696, "api_type": "wasi-http", "status": 1}
```

```sh
# Set BINARY_ID to the id from the upload response, then create the app
export BINARY_ID=4696

curl -sX POST 'https://api.gcore.com/fastedge/v1/apps' \
  -H "Authorization: APIKey $GCORE_API_KEY" \
  -H 'Content-Type: application/json' \
  -d "{\"name\": \"my-outbound-fetch\", \"binary\": $BINARY_ID, \"status\": 1}"
```

The URL from the create-app response becomes active within a few seconds:

```sh
curl https://my-outbound-fetch-1000503.fastedge.app/
```

The component fetches from the upstream API and returns a shaped response:

```json
{
  "limit": 30,
  "skip": 0,
  "total": 5,
  "users": [
    {
      "id": 1,
      "name": "Leanne Graham",
      "username": "Bret",

      "phone": "1-770-736-8031 x56442",
```

The outbound call happens on every request, from every edge node that handles traffic for this app — for data that changes infrequently, storing the upstream response in a [FastEdge KV store](/fastedge/kv-stores/manage-kv-store) eliminates that per-request latency after the first fetch.

## Cleanup

To delete an app, use its `id` from the create-app response:

```sh
curl -sX DELETE "https://api.gcore.com/fastedge/v1/apps/{APP_ID}" \
  -H "Authorization: APIKey $GCORE_API_KEY"
```

Deleting the app does not remove the binary — to remove it as well:

```sh
curl -sX DELETE "https://api.gcore.com/fastedge/v1/binaries/{BINARY_ID}" \
  -H "Authorization: APIKey $GCORE_API_KEY"
```

A binary referenced by an active application cannot be deleted — remove the app first.
