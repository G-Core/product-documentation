If Object Storage or SFTP storage returns an error, first check the [Gcore Status Page](https://status.gcore.com) for any active incidents or maintenance that may be causing the issue.

## HTTP 404

A 404 error means the storage cannot find the requested file. Two common causes:

1. The file hasn't been uploaded yet — confirm the file exists via the portal or an S3-compatible client.
2. The request URL is malformed — Object Storage files are accessed as `http(s)://<bucket-name>.<hostname>`, where the hostname for each location is shown in the **Details** section of the storage in the portal. See the [storage access guide](/storage/request-content-directly-from-the-storage) for the full URL schema and available hostnames.

## HTTP 403

A 403 error means the requested file or bucket is not publicly accessible. By default, Object Storage buckets and files are private — external requests return a 403 error.

To allow public access, set an ACL (Access Control List) rule on the bucket or file. Both AWS CLI and s3cmd support this operation:

**AWS CLI:**

```sh
aws s3api put-object-acl --bucket my_bucket --key file.jpg --acl public-read --endpoint-url=https://s3-ed1.cloud.gcore.lu
```

Where `my_bucket` is the bucket name, `file.jpg` is the file name, and `s3-ed1.cloud.gcore.lu` is the service URL for the storage location.

**s3cmd:**

```sh
s3cmd setacl --acl-public s3://my_bucket/file.jpg
```

Where `my_bucket` is the bucket name and `file.jpg` is the file name.
