A load balancer distributes incoming requests across two or more backend VMs, eliminating single points of failure. This guide creates two VMs, a load balancer with a public VIP, a listener, a pool, and adds both VMs as members in 8 API calls.

<Info>
A permanent [API token](/developer-tools/rest-api/authentication) is required, along with a project ID and region ID. Retrieve them from the [Gcore Customer Portal](https://portal.gcore.com) or by calling <code>GET&nbsp;/cloud/v1/projects</code> and <code>GET&nbsp;/cloud/v1/regions</code>.
</Info>

```bash
export GCORE_API_KEY="{YOUR_API_KEY}"
export PROJECT_ID="{YOUR_PROJECT_ID}"
export REGION_ID="{YOUR_REGION_ID}"
```

## Step 1. Create the backend VMs

Create two VMs that will serve as backends. Both need public interfaces so the load balancer can reach them.

| Parameter | Required | Description |
|-----------|----------|-------------|
| `interfaces[].type` | Yes | `"external"` assigns a public IP  -  the load balancer routes to this IP. |

<Tabs>
  <Tab title="curl">
    ```bash
    # Create VM1 (repeat with a different name for VM2)
    curl -X POST "https://api.gcore.com/cloud/v2/instances/$PROJECT_ID/$REGION_ID" \
      -H "Authorization: APIKey $GCORE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{
        "name": "backend-1",
        "flavor": "g1-standard-2-4",
        "ssh_key_name": "my-key",
        "interfaces": [{"type": "external"}],
        "volumes": [{
          "source": "image",
          "image_id": "{IMAGE_ID}",
          "size": 20,
          "boot_index": 0
        }]
      }'
    ```
  </Tab>
  <Tab title="Python SDK">
    ```python
    import os, time
    from gcore import Gcore

    client = Gcore(api_key=os.environ["GCORE_API_KEY"])

    def create_vm(name):
        r = client.cloud.instances.create(
            project_id=int(os.environ["PROJECT_ID"]),
            region_id=int(os.environ["REGION_ID"]),
            name=name,
            flavor="g1-standard-2-4",
            ssh_key_name="my-key",
            interfaces=[{"type": "external"}],
            volumes=[{"source": "image", "image_id": os.environ["IMAGE_ID"],
                      "size": 20, "boot_index": 0}],
        )
        task = client.cloud.tasks.poll(r.tasks[0], polling_interval_seconds=5,
                                       polling_timeout_seconds=120)
        vm_id = task.created_resources.instances[0]
        inst = client.cloud.instances.get(
            instance_id=vm_id,
            project_id=int(os.environ["PROJECT_ID"]),
            region_id=int(os.environ["REGION_ID"]),
        )
        pub_ip = next(iter(inst.addresses.values()))[0].addr
        return vm_id, pub_ip

    vm1_id, vm1_ip = create_vm("backend-1")
    vm2_id, vm2_ip = create_vm("backend-2")
    print(f"VM1: {vm1_ip}, VM2: {vm2_ip}")
    ```
  </Tab>
</Tabs>

Poll both tasks until `FINISHED`  -  see [Async operations](/developer-tools/rest-api/async-operations). Save the public IP of each VM; it is used when adding members to the pool.

```bash
export VM1_IP="{VM1_PUBLIC_IP}"
export VM2_IP="{VM2_PUBLIC_IP}"
```

## Step 2. List load balancer flavors

The LB flavor determines the vCPU, RAM, and network throughput of the load balancer instance.

<Tabs>
  <Tab title="curl">
    ```bash
    curl "https://api.gcore.com/cloud/v1/lbflavors/$PROJECT_ID/$REGION_ID" \
      -H "Authorization: APIKey $GCORE_API_KEY"
    ```
  </Tab>
  <Tab title="Python SDK">
    ```python
    flavors = client.cloud.load_balancers.flavors.list(
        project_id=int(os.environ["PROJECT_ID"]),
        region_id=int(os.environ["REGION_ID"]),
    )
    for f in flavors.results:
        print(f.flavor_name, f.vcpus, "vCPU /", f.ram // 1024, "GB")
    ```
  </Tab>
</Tabs>

```json
{
  "results": [
    {"flavor_id": "fca2fc7c-...", "flavor_name": "lb1-1-2", "vcpus": 1, "ram": 2048},
    {"flavor_id": "b5e956d8-...", "flavor_name": "lb1-2-4", "vcpus": 2, "ram": 4096}
  ]
}
```

```bash
export LB_FLAVOR="lb1-1-2"
```

<Info>
The `flavor` field in the create request accepts the `flavor_name` string (`lb1-1-2`), not the UUID `flavor_id`.
</Info>

## Step 3. Create the load balancer

Creating the load balancer without specifying a network assigns a public VIP automatically.

| Parameter | Required | Description |
|-----------|----------|-------------|
| `name` | Yes | Display name for the load balancer. |
| `flavor` | Yes | The `flavor_name` from Step 2. |

<Tabs>
  <Tab title="curl">
    ```bash
    curl -X POST "https://api.gcore.com/cloud/v1/loadbalancers/$PROJECT_ID/$REGION_ID" \
      -H "Authorization: APIKey $GCORE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{
        "name": "my-lb",
        "flavor": "'"$LB_FLAVOR"'"
      }'
    ```
  </Tab>
  <Tab title="Python SDK">
    ```python
    r = client.cloud.load_balancers.create(
        project_id=int(os.environ["PROJECT_ID"]),
        region_id=int(os.environ["REGION_ID"]),
        name="my-lb",
        flavor=os.environ["LB_FLAVOR"],
    )
    task = client.cloud.tasks.poll(r.tasks[0], polling_interval_seconds=10,
                                   polling_timeout_seconds=300)
    lb_id = task.created_resources.loadbalancers[0]
    ```
  </Tab>
</Tabs>

```json
{
  "tasks": ["d4a4b10e-..."]  // save as LB_TASK_ID
}
```

Poll every 10 seconds until `FINISHED`  -  load balancer provisioning takes 2-3 minutes. Retrieve the VIP address:

```bash
curl "https://api.gcore.com/cloud/v1/loadbalancers/$PROJECT_ID/$REGION_ID/$LB_ID" \
  -H "Authorization: APIKey $GCORE_API_KEY"
```

```json
{
  "id": "d19e4a23-...",
  "provisioning_status": "ACTIVE",
  "operating_status": "ONLINE",
  "vip_address": "85.234.84.40"   // public IP clients connect to
}
```

```bash
export LB_ID="{LB_ID}"
```

## Step 4. Create a listener

A listener watches for incoming connections on a specific protocol and port, then forwards them to a pool.

| Parameter | Required | Description |
|-----------|----------|-------------|
| `load_balancer_id` | Yes | The LB ID from Step 3. |
| `protocol` | Yes | `HTTP`, `TCP`, `UDP`, or `TERMINATED_HTTPS`. |
| `protocol_port` | Yes | Port to listen on (1-65535). |

<Tabs>
  <Tab title="curl">
    ```bash
    curl -X POST "https://api.gcore.com/cloud/v1/lblisteners/$PROJECT_ID/$REGION_ID" \
      -H "Authorization: APIKey $GCORE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{
        "name": "http-listener",
        "load_balancer_id": "'"$LB_ID"'",
        "protocol": "HTTP",
        "protocol_port": 80
      }'
    ```
  </Tab>
  <Tab title="Python SDK">
    ```python
    r = client.cloud.load_balancers.listeners.create(
        project_id=int(os.environ["PROJECT_ID"]),
        region_id=int(os.environ["REGION_ID"]),
        load_balancer_id=lb_id,
        name="http-listener",
        protocol="HTTP",
        protocol_port=80,
    )
    task = client.cloud.tasks.poll(r.tasks[0], polling_interval_seconds=5,
                                   polling_timeout_seconds=120)
    listener_id = task.created_resources.listeners[0]
    ```
  </Tab>
</Tabs>

```json
{
  "tasks": ["f1e2d3c4-..."]  // save as LISTENER_TASK_ID
}
```

```bash
export LISTENER_ID="{LISTENER_ID}"
```

## Step 5. Create a pool

A pool groups the backend VMs. The load balancer distributes requests across pool members according to the chosen algorithm.

| Parameter | Required | Description |
|-----------|----------|-------------|
| `listener_id` | Yes | Attaches the pool to the listener from Step 4. |
| `load_balancer_id` | Yes | The LB ID. |
| `lb_algorithm` | Yes | `ROUND_ROBIN`, `LEAST_CONNECTIONS`, or `SOURCE_IP`. |
| `protocol` | Yes | Must match the listener protocol. |

<Tabs>
  <Tab title="curl">
    ```bash
    curl -X POST "https://api.gcore.com/cloud/v1/lbpools/$PROJECT_ID/$REGION_ID" \
      -H "Authorization: APIKey $GCORE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{
        "name": "http-pool",
        "lb_algorithm": "ROUND_ROBIN",
        "protocol": "HTTP",
        "listener_id": "'"$LISTENER_ID"'",
        "load_balancer_id": "'"$LB_ID"'"
      }'
    ```
  </Tab>
  <Tab title="Python SDK">
    ```python
    r = client.cloud.load_balancers.pools.create(
        project_id=int(os.environ["PROJECT_ID"]),
        region_id=int(os.environ["REGION_ID"]),
        name="http-pool",
        lb_algorithm="ROUND_ROBIN",
        protocol="HTTP",
        listener_id=listener_id,
        load_balancer_id=lb_id,
    )
    task = client.cloud.tasks.poll(r.tasks[0], polling_interval_seconds=5,
                                   polling_timeout_seconds=120)
    pool_id = task.created_resources.pools[0]
    ```
  </Tab>
</Tabs>

```bash
export POOL_ID="{POOL_ID}"
```

## Step 6. Add VMs as pool members

Add each backend VM to the pool by specifying its IP address and the port where the application listens.

| Parameter | Required | Description |
|-----------|----------|-------------|
| `address` | Yes | The VM's public IP address (from Step 1). |
| `protocol_port` | Yes | The port the application is listening on inside the VM. |
| `instance_id` | No | Helps the load balancer identify the VM for health monitoring. |

<Tabs>
  <Tab title="curl">
    ```bash
    # Add VM1 (repeat for VM2 with its IP)
    curl -X POST "https://api.gcore.com/cloud/v1/lbpools/$PROJECT_ID/$REGION_ID/$POOL_ID/member" \
      -H "Authorization: APIKey $GCORE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{
        "address": "'"$VM1_IP"'",
        "protocol_port": 80
      }'
    ```
  </Tab>
  <Tab title="Python SDK">
    ```python
    for vm_id, vm_ip in [(vm1_id, vm1_ip), (vm2_id, vm2_ip)]:
        r = client.cloud.load_balancers.pools.members.create(
            pool_id=pool_id,
            project_id=int(os.environ["PROJECT_ID"]),
            region_id=int(os.environ["REGION_ID"]),
            address=vm_ip,
            protocol_port=80,
            instance_id=vm_id,
        )
        client.cloud.tasks.poll(r.tasks[0], polling_interval_seconds=5,
                                polling_timeout_seconds=60)
        print(f"Added {vm_ip} to pool")
    ```
  </Tab>
</Tabs>

<Info>
Members show `operating_status: NO_MONITOR` when no health monitor is configured  -  this is expected. The load balancer still distributes traffic. Add a health monitor to the pool to enable automatic member removal when a backend becomes unhealthy.
</Info>

## Step 7. Verify the load balancer

After both members are added, confirm the load balancer is ready:

```bash
curl "https://api.gcore.com/cloud/v1/loadbalancers/$PROJECT_ID/$REGION_ID/$LB_ID" \
  -H "Authorization: APIKey $GCORE_API_KEY"
```

```json
{
  "provisioning_status": "ACTIVE",
  "operating_status": "ONLINE",
  "vip_address": "85.234.84.40",
  "listeners": [{"id": "9ae94e55-..."}],
  "pools": [{"id": "21a314c4-..."}]
}
```

Incoming HTTP traffic to `85.234.84.40:80` is now distributed across both VMs using round-robin.

## Clean up

Delete in dependency order: members are removed automatically with the pool, the pool is removed with the LB.

1. Delete the load balancer (removes listeners, pool, and members):

```bash
curl -X DELETE \
  "https://api.gcore.com/cloud/v1/loadbalancers/$PROJECT_ID/$REGION_ID/$LB_ID" \
  -H "Authorization: APIKey $GCORE_API_KEY"
```

2. Delete the backend VMs:

```bash
# Delete VM1
curl -X DELETE \
  "https://api.gcore.com/cloud/v1/instances/$PROJECT_ID/$REGION_ID/$VM1_ID?volumes=$VM1_BOOT_VOL" \
  -H "Authorization: APIKey $GCORE_API_KEY"

# Delete VM2
curl -X DELETE \
  "https://api.gcore.com/cloud/v1/instances/$PROJECT_ID/$REGION_ID/$VM2_ID?volumes=$VM2_BOOT_VOL" \
  -H "Authorization: APIKey $GCORE_API_KEY"
```

Poll both delete tasks to `FINISHED` before deprovisioning is complete.
