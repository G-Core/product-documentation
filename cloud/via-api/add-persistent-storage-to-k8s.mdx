Gcore Managed Kubernetes clusters ship with four StorageClasses backed by Cinder block storage. Two kubectl manifests provision a PersistentVolumeClaim and deploy a PostgreSQL StatefulSet with a dedicated 5 GiB volume; the final steps confirm data survives a pod restart and demonstrate online volume resize.

<Info>
A running cluster with kubectl configured is required. The [deployment guide](/cloud/via-api/deploy-kubernetes-cluster) covers provisioning a cluster and downloading the kubeconfig.
</Info>

If the kubeconfig is not at the default path (`~/.kube/config`), set the environment variable:

```bash
export KUBECONFIG="/path/to/your/kubeconfig"
```

## Step 1. Check available StorageClasses

Every Gcore Managed Kubernetes cluster comes with four pre-installed StorageClasses — no manual provisioner setup required.

```bash
kubectl get storageclass
```

```
NAME                            PROVISIONER                RECLAIMPOLICY   VOLUMEBINDINGMODE   ALLOWVOLUMEEXPANSION
csi-sc-cinderplugin (default)   cinder.csi.gcorelabs.com   Delete          Immediate           true
ssd-hiiops                      cinder.csi.gcorelabs.com   Delete          Immediate           true
ssd-lowlatency                  cinder.csi.gcorelabs.com   Delete          Immediate           true
standard                        cinder.csi.gcorelabs.com   Delete          Immediate           true
```

Each StorageClass maps to a Cinder volume type with different performance characteristics:

| StorageClass | Disk type | Use case |
|---|---|---|
| `csi-sc-cinderplugin` (default) | Standard SSD | General-purpose workloads |
| `standard` | Standard SSD | General-purpose workloads |
| `ssd-hiiops` | High-IOPS SSD (60 IOPS/GiB, 9,000 IOPS max) | Databases, transaction-heavy workloads |
| `ssd-lowlatency` | Low-latency SSD (avg. 300 µs) | Real-time data processing |

All four support online volume expansion (`ALLOWVOLUMEEXPANSION: true`). The `ReclaimPolicy: Delete` setting means the underlying Cinder volume is deleted when the PVC is deleted.

## Step 2. Deploy a StatefulSet with a persistent volume

A StatefulSet's `volumeClaimTemplates` field instructs Kubernetes to provision a dedicated PVC per replica, named after the template combined with the pod ordinal (for example, `postgres-storage-postgres-0`). The same PVC re-attaches to the same pod after every restart, making the volume persistent across the pod lifecycle.

The key fields in the volume claim template are:

| Parameter | Required | Description |
|---|---|---|
| `storageClassName` | Yes | StorageClass from Step 1. Omit to use the cluster default (`csi-sc-cinderplugin`) |
| `accessModes: ReadWriteOnce` | Yes | The volume mounts to one node at a time — required for Cinder block storage |
| `resources.requests.storage` | Yes | Requested volume size in GiB. Minimum 1 GiB |
| `serviceName` | Yes | Name of the headless Service defined in the same manifest. StatefulSets use it to assign stable DNS records to each pod |

<Warning>
Set `PGDATA` to a subdirectory of the mount point (`/var/lib/postgresql/data/pgdata`, not `/var/lib/postgresql/data`). PostgreSQL refuses to initialize in a directory that already contains files — the volume root has a `lost+found` directory created by the ext4 filesystem.
</Warning>

Save the following manifest as `postgres-statefulset.yaml`:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: postgres
spec:
  clusterIP: None
  selector:
    app: postgres
  ports:
    - port: 5432
---
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  selector:
    matchLabels:
      app: postgres
  serviceName: postgres
  replicas: 1
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: postgres:16
          env:
            - name: POSTGRES_PASSWORD
              value: "{YOUR_PASSWORD}"
            - name: PGDATA
              value: /var/lib/postgresql/data/pgdata
          ports:
            - containerPort: 5432
          volumeMounts:
            - name: postgres-storage
              mountPath: /var/lib/postgresql/data
  volumeClaimTemplates:
    - metadata:
        name: postgres-storage
      spec:
        accessModes:
          - ReadWriteOnce
        storageClassName: csi-sc-cinderplugin
        resources:
          requests:
            storage: 5Gi
```

Apply the manifest:

```bash
kubectl apply -f postgres-statefulset.yaml
```

```
service/postgres created
statefulset.apps/postgres created
```

Confirm the pod is running and the PVC is bound:

```bash
kubectl get pods,pvc
```

```
NAME             READY   STATUS    RESTARTS   AGE
pod/postgres-0   1/1     Running   0          35s

NAME                                                STATUS   VOLUME                                     CAPACITY
persistentvolumeclaim/postgres-storage-postgres-0   Bound    pvc-d53fc2b9-3b08-4730-bdc9-fc5c1c145cb6   5Gi
```

## Step 3. Verify persistence

Write a row to the database, delete the pod, wait for the StatefulSet to restart it, and confirm the row is still present:

```bash
kubectl exec postgres-0 -- psql -U postgres -c "CREATE DATABASE testdb;"
kubectl exec postgres-0 -- psql -U postgres -d testdb -c \
  "CREATE TABLE items (id serial, name text); INSERT INTO items (name) VALUES ('test-data');"

kubectl delete pod postgres-0

kubectl wait --for=condition=Ready pod/postgres-0 --timeout=60s
kubectl exec postgres-0 -- psql -U postgres -d testdb -c "SELECT * FROM items;"
```

```
 id |   name
----+-----------
  1 | test-data
(1 row)
```

The row survives the restart because the StatefulSet re-attaches the PVC `postgres-storage-postgres-0` to the new pod instance.

## Step 4. Resize the volume

To increase the volume capacity, patch the PVC's storage request. Decreasing the size is not supported.

Save the following patch as `pvc-resize.yaml`:

```yaml
spec:
  resources:
    requests:
      storage: 10Gi
```

Apply the patch:

```bash
kubectl patch pvc postgres-storage-postgres-0 --patch-file pvc-resize.yaml
```

The Cinder CSI driver resizes the block volume and expands the filesystem while the pod continues running. Confirm the new capacity:

```bash
kubectl get pvc postgres-storage-postgres-0
```

```
NAME                          STATUS   VOLUME                                     CAPACITY   ACCESS MODES
postgres-storage-postgres-0   Bound    pvc-d53fc2b9-3b08-4730-bdc9-fc5c1c145cb6   10Gi       RWO
```
