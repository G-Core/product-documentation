A Bare Metal server is a physical machine allocated entirely to a single tenant, with no hypervisor layer. All CPU, RAM, and storage resources are dedicated - not shared with other users.

<Info>
  **Info**

  Bare Metal servers require a quota increase before creation. Select a region and server type in the creation form, then click **Send quota request** in the **Quota Increase Required** block. Requests are typically reviewed within 5 minutes, with a 95% approval rate.
</Info>

## Step 1. Open the creation page

In the [Gcore Customer Portal](https://portal.gcore.com/accounts/reports/dashboard), navigate to **Cloud** > **Bare Metal** and click **Create Bare Metal**.

<Frame>
  ![Bare metal page in Gcore Customer Portal](/images/docs/cloud/bare-metal-servers/create-a-bare-metal-server/bare-metal-page.png)
</Frame>

## Step 2. Select a region

Choose the region where the server will be deployed.

## Step 3. Select an image

Select the architecture - **x86-64** or **ARM** - then choose an OS image from one of three tabs:

- **Distributions** - prepared OS templates (Ubuntu, CentOS, Debian, Windows, and others).
- **Marketplace** - pre-configured application images.
- **Custom Images** - images previously uploaded to the account. Only images built for Bare Metal servers are compatible; standard VM images do not work. Build requirements are covered in [image upload](/cloud/images/upload-an-image-to-the-storage).

<Frame>
  ![A menu with available OS images](/images/docs/cloud/bare-metal-servers/create-a-bare-metal-server/select-image.png)
</Frame>

## Step 4. Select a server type

Server types are grouped into three categories:

- **High frequency** - high clock-speed CPUs for latency-sensitive workloads.
- **Infrastructure** - high core count and NVMe storage for compute-intensive tasks.
- **Storage** - large-capacity drives for data-heavy workloads.

<Frame>
  ![A menu with available server types](/images/docs/cloud/bare-metal-servers/create-a-bare-metal-server/server-type.png)
</Frame>

All types require a quota. After selecting a type, a **Quota Increase Required** block appears with a **Send quota request** button. The remaining sections of the form unlock once the quota is approved.

## Step 5. Configure network interfaces

After quota approval, configure network interfaces. Up to six networks can be attached. An external network cannot be added after the server is created.

1. Select the **Network type**: **Public**, **Private**, or **Dedicated public**.

2. In the **IP Family** section, select the IP version: **IPv4**, **IPv6**, or **Dual (IPv4 + IPv6)**.

3. (Optional) Enable **Use reserved IP** to assign a [reserved IP address](/cloud/networking/ip-address/create-and-configure-a-reserved-ip-address) to the server.

4. (Optional) Enable **Advanced DDoS Protection** for enhanced security.

<Frame>
  ![Network interfaces configuration](/images/docs/cloud/bare-metal-servers/create-a-bare-metal-server/bm-network-section.png)
</Frame>

For a **private** interface, also configure a network and subnetwork:

<Info>
  **Info**

  With both a public and a private interface, disable the default gateway on the private subnetwork and assign a floating IP to the private interface to avoid routing conflicts.
</Info>

<Tabs>
  <Tab title="Configure a network">
    Select an existing network from the dropdown, or click the **Private network** radio button to create a new one:

    1. Enter the network name.

    2. Confirm that the **Bare Metal network** toggle is enabled - this is required to connect Bare Metal servers to the network.

    3. (Optional) Enable **Add tags** to attach metadata to the network.

    4. Click **Create network**.

    <Frame>
      ![A menu with available network settings and highlighted reserved ip toggle](/images/docs/cloud/bare-metal-servers/create-a-bare-metal-server/create-network-dialog.png)
    </Frame>
  </Tab>
  <Tab title="Configure a subnetwork">
    <Info>
      **Info**

      With multiple subnetworks, set only one as [routable](/cloud/networking/create-and-manage-a-subnetwork#set-up-the-default-gateway). Multiple routable subnetworks cause default gateway conflicts that prevent connecting to the server.
    </Info>

    Select an existing subnetwork from the dropdown. Choosing **Automatic** assigns the subnetwork with the most available resources.

    To create a new subnetwork, click **Add a new subnetwork** and follow the steps in [Create a subnetwork](/cloud/networking/create-and-manage-a-subnetwork#create-a-subnetwork).
  </Tab>
</Tabs>

A dedicated public network provides an individual address pool, allowing multiple public IPs on a single network port - useful for high-traffic applications, load distribution, or virtualization scenarios where each workload needs a distinct public IP.

### Interface order

Bare Metal servers use LACP bonding: all physical ports are aggregated into a single bond (`bond0`). Each sub-interface added in the creation form becomes a logical port on top of that bond.

- **Sub-Interface 1** (public) maps to `bond0`.
- **Sub-Interface 2** (private) maps to a VLAN sub-interface named `bond0.<vlan_id>` (for example, `bond0.2853`).

After the server is created, the **Networking** tab shows the sub-interfaces in the same order as they were added during setup. Running `ip a` in the OS confirms the mapping:

<Frame>
  ![Diagram showing how Sub-Interface 1 and Sub-Interface 2 in the creation form map to the Networking tab entries and to bond0 and bond0.2853 in the OS](/images/docs/cloud/bare-metal-servers/create-a-bare-metal-server/bm-network-interface-order-diagram.png)
</Frame>

```bash
$ ip a
1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536 ...
2: enp1s0f0: <BROADCAST,MULTICAST,SLAVE,UP,LOWER_UP> mtu 1500 ... master bond0
4: enp1s0f1: <BROADCAST,MULTICAST,SLAVE,UP,LOWER_UP> mtu 1500 ... master bond0
5: bond0: <BROADCAST,MULTICAST,MASTER,UP,LOWER_UP> mtu 1500 qdisc noqueue state UP
    link/ether 7c:c2:55:6c:ad:b0 brd ff:ff:ff:ff:ff:ff
    inet 85.234.84.162/24 brd 85.234.84.255 scope global bond0
6: bond0.2853@bond0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc noqueue state UP
    link/ether 7c:c2:55:6c:ad:b0 brd ff:ff:ff:ff:ff:ff
    inet 10.0.0.125/24 brd 10.0.0.255 scope global dynamic bond0.2853
```

Sub-Interface 1 (`bond0`) is the public interface; Sub-Interface 2 (`bond0.2853`) is the private interface. The order set at creation determines routing priority: the first sub-interface has higher priority for outbound traffic than subsequent ones. The sub-interface order cannot be changed after the server is created.

## Step 6. Configure access

Select an SSH key for Linux servers, or set a password for Windows servers.

<Frame>
  ![A menu with SSH keys settings settings](/images/docs/cloud/bare-metal-servers/create-a-bare-metal-server/ssh-keys.png)
</Frame>

To add a new key, follow the steps in [Connect via SSH](/cloud/bare-metal-servers/connect-to-your-bare-metal-server-via-ssh).

For Windows servers, the password must be 8-16 characters and may contain Latin letters (`a-zA-Z`), digits (`0-9`), and special characters (`!#$%&'()*+,-./:;<=>?@[]^_{|}~`). Connect via [the portal console](/cloud/virtual-instances/connect/connect-to-your-instance-via-control-panel) or RDP.

<Frame>
  ![A menu with access settings](/images/docs/cloud/bare-metal-servers/create-a-bare-metal-server/access.png)
</Frame>

## Step 7. Configure additional options

In the **Additional options** section, optionally provide a `cloud-init` script in the **User data** field to automate initial configuration - install packages, create users, or run commands on first boot.

<Frame>
  ![A menu with user data settings](/images/docs/cloud/bare-metal-servers/create-a-bare-metal-server/user-data.png)
</Frame>

## Step 8. Set quantity and name

Specify the number of servers (up to 10, subject to quota) and the name template. Click **Create Server** - the server provisions in the cloud.

<Frame>
  ![A menu with an option to choose a number of servers](/images/docs/cloud/bare-metal-servers/create-a-bare-metal-server/servers-number.png)
</Frame>

## Multi-bond network configuration

Bare Metal servers support multiple network bonds, where each bond is a group of LACP-aggregated physical ports. With a multi-bond setup, traffic for different networks is isolated at the hardware level, which improves availability and throughput. Multi-bond configuration is available on request - contact [Gcore support](https://gcore.com/contact-us) to enable it.

After the server is deployed, open the server details and go to the **Networking** tab. Each bond appears as a separate interface block labeled **Interface N (Trunk Interface N)**.

<Frame>
  ![Bare Metal server Networking tab showing multi-bond configuration with Interface 1 (Trunk Interface 1) and Interface 2 (Trunk Interface 2), each with one sub-interface](/images/docs/cloud/bare-metal-servers/create-a-bare-metal-server/bm-multi-bond-networking-tab.png)
</Frame>

Each interface block shows the sub-interfaces assigned to that bond - a sub-interface is the logical port that carries traffic for a specific network. The table for each sub-interface includes the IP address, network, subnetwork, MAC address, CIDR, and floating IP assignment.

To add a sub-interface to a bond, click **Add Sub-Interface** in the corresponding interface block and configure the network settings.

## Limitations

- External volumes cannot be attached to Bare Metal servers.
- Volume configuration cannot be changed after provisioning.
- No more than six networks can be attached.
- After deployment, private network interfaces can only be attached or detached manually via the OS.

<MethodSection id="api" label="REST API">

A Bare Metal server is a physical machine allocated entirely to one tenant - no hypervisor, no shared resources. Provisioning takes 4-5 minutes.

<Info>
A permanent [API token](/developer-tools/rest-api/authentication) is required, along with a project ID and region ID. Retrieve them from the [Gcore Customer Portal](https://portal.gcore.com) or by calling <code>GET&nbsp;/cloud/v1/projects</code> and <code>GET&nbsp;/cloud/v1/regions</code>.
</Info>

Open a bash terminal and set these as environment variables before running the examples:

```bash
export GCORE_API_KEY="{YOUR_API_KEY}"
export PROJECT_ID="{YOUR_PROJECT_ID}"
export REGION_ID="{YOUR_REGION_ID}"
```

<ApiH2>Step 1. List available flavors</ApiH2>

Bare Metal flavors describe the physical hardware: CPU model, core count, RAM, local disk, and network speed.

<Tabs>
  <Tab title="curl">
    ```bash
    curl "https://api.gcore.com/cloud/v1/bmflavors/$PROJECT_ID/$REGION_ID?disabled=false&include_capacity=true" \
      -H "Authorization: APIKey $GCORE_API_KEY"
    ```
  </Tab>
  <Tab title="Python SDK">
    ```python
    flavors = client.cloud.baremetal.flavors.list(
        project_id=int(os.environ["PROJECT_ID"]),
        region_id=int(os.environ["REGION_ID"]),
        disabled=False,
        include_capacity=True,
    )
    for f in flavors.results:
        print(f.flavor_id, f.hardware_description)
    ```
  </Tab>
</Tabs>

The API returns:

```json
{
  "results": [
    {
      "flavor_id": "bm1a-hf-1x7950x-128-2x960gb-2x10nic",
      "vcpus": 32,
      "ram": 131072,
      "hardware_description": {
        "cpu": "1x AMD Ryzen 7950X 16C/32T 4.5 GHz",
        "ram": "128 GB",
        "disk": "2x 960GB NVMe",
        "network": "2x 10GbE"
      },
      "capacity": 1
    }
  ]
}
```

`capacity` shows the number of available physical machines in the region - a value of `0` means the flavor is temporarily unavailable.

```bash
export FLAVOR_ID="bm1a-hf-1x7950x-128-2x960gb-2x10nic"
```

<ApiH2>Step 2. Find a bare metal OS image</ApiH2>

Bare Metal servers use a separate image pool from virtual machines. BM images are identified by an `-ironic` suffix in the name and `"is_baremetal": true` in the response. They are not returned by the standard images list - retrieve a known image ID directly or identify it from the Customer Portal under **Bare Metal > Distributions**.

<Tabs>
  <Tab title="curl">
    ```bash
    # Retrieve a known BM image by ID to confirm it is available in the region
    curl "https://api.gcore.com/cloud/v1/images/$PROJECT_ID/$REGION_ID/{IMAGE_ID}" \
      -H "Authorization: APIKey $GCORE_API_KEY"

    # Validate that the image is compatible with the chosen flavor
    curl -X POST "https://api.gcore.com/cloud/v1/bminstances/$PROJECT_ID/$REGION_ID/available_flavors" \
      -H "Authorization: APIKey $GCORE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"image_id": "{IMAGE_ID}"}'
    ```
  </Tab>
  <Tab title="Python SDK">
    ```python
    # Validate image against available flavors
    compatible = client.cloud.baremetal.servers.available_flavors(
        project_id=int(os.environ["PROJECT_ID"]),
        region_id=int(os.environ["REGION_ID"]),
        image_id=os.environ["IMAGE_ID"],
    )
    for f in compatible.results:
        print(f.flavor_id)
    ```
  </Tab>
</Tabs>

Image confirmation response:

```json
{
  "id": "8cab6f28-09ca-4201-b3f7-23c7893f4bd6",
  "name": "ubuntu-24.04-x64-ironic",
  "is_baremetal": true,
  "visibility": "public",
  "os_distro": "ubuntu",
  "os_version": "24.04"
}
```

```bash
export IMAGE_ID="{IMAGE_ID}"
```

<ApiH2>Step 3. Create the bare metal server</ApiH2>

The request body requires a flavor, image, SSH key, and at least one network interface.

<Info>
An SSH key must already exist in the project. Add one with <code>POST&nbsp;/cloud/v1/ssh_keys/{project_id}</code> as described in the [VM deployment guide](/cloud/virtual-instances/create-an-instance#step-1-add-an-ssh-key).
</Info>

| Parameter | Required | Description |
|-----------|----------|-------------|
| `flavor` | Yes | Flavor ID from Step 1. |
| `image_id` | Yes | BM image ID from Step 2. Cannot use standard VM images. |
| `ssh_key_name` | Yes | Name of an existing SSH key in the project. |
| `interfaces[].type` | Yes | `"external"` assigns a public IP. `"subnet"` attaches to a private network. |

<Tabs>
  <Tab title="curl">
    ```bash
    curl -X POST "https://api.gcore.com/cloud/v1/bminstances/$PROJECT_ID/$REGION_ID" \
      -H "Authorization: APIKey $GCORE_API_KEY" \
      -H "Content-Type: application/json" \
      -d "{
        \"flavor\": \"$FLAVOR_ID\",
        \"image_id\": \"$IMAGE_ID\",
        \"ssh_key_name\": \"my-key\",
        \"interfaces\": [{\"type\": \"external\"}],
        \"name_templates\": [\"{REGION_PREFIX}-{ip_octets}\"]
      }"
    ```
  </Tab>
  <Tab title="Python SDK">
    ```python
    result = client.cloud.baremetal.servers.create(
        project_id=int(os.environ["PROJECT_ID"]),
        region_id=int(os.environ["REGION_ID"]),
        flavor=os.environ["FLAVOR_ID"],
        image_id=os.environ["IMAGE_ID"],
        ssh_key_name="my-key",
        interfaces=[{"type": "external"}],
        name_templates=["{REGION_PREFIX}-{ip_octets}"],
    )
    task_id = result.tasks[0]
    ```
  </Tab>
</Tabs>

The API returns:

```json
{
  "tasks": ["d4a4b10e-5d22-4c23-8e09-f7a3e1c1a999"]
}
```

<Info>
The `name_templates` field follows a region-specific pattern. If the creation returns "name templates are not allowed", the error message lists the allowed values for that region - for example, `"lux-b2-{ip_octets}"` in Luxembourg-3.
</Info>

<ApiH2>Step 4. Wait for provisioning</ApiH2>

Bare Metal provisioning takes 4-5 minutes. Poll <code>GET&nbsp;/cloud/v1/tasks/{task_id}</code> every 30 seconds until `state` is `FINISHED`.

<Tabs>
  <Tab title="curl">
    ```bash
    curl "https://api.gcore.com/cloud/v1/tasks/$TASK_ID" \
      -H "Authorization: APIKey $GCORE_API_KEY"
    ```
  </Tab>
  <Tab title="Python SDK">
    ```python
    import time

    while True:
        task = client.cloud.tasks.get(task_id)
        if task.state == "FINISHED":
            server_id = task.created_resources.instances[0]
            break
        if task.state == "ERROR":
            raise RuntimeError(task.error)
        time.sleep(30)
    ```
  </Tab>
</Tabs>

The API returns:

```json
{
  "state": "FINISHED",
  "created_on": "2026-05-21T13:23:13",
  "finished_on": "2026-05-21T13:27:40",
  "created_resources": {
    "instances": ["a3b4c5d6-e7f8-4abc-def0-123456789012"]
  }
}
```

```bash
export SERVER_ID="{SERVER_ID}"
```

<ApiH2>Step 5. Get the server details</ApiH2>

Retrieve the server to confirm it is `ACTIVE` and find the assigned public IP address.

<Tabs>
  <Tab title="curl">
    ```bash
    curl "https://api.gcore.com/cloud/v1/bminstances/$PROJECT_ID/$REGION_ID/$SERVER_ID" \
      -H "Authorization: APIKey $GCORE_API_KEY"
    ```
  </Tab>
  <Tab title="Python SDK">
    ```python
    server = client.cloud.baremetal.servers.get(
        server_id=os.environ["SERVER_ID"],
        project_id=int(os.environ["PROJECT_ID"]),
        region_id=int(os.environ["REGION_ID"]),
    )
    for network, ifaces in server.addresses.items():
        print(network, ifaces[0]["addr"])
    ```
  </Tab>
</Tabs>

The API returns:

```json
{
  "id": "a3b4c5d6-e7f8-4abc-def0-123456789012",
  "name": "lux-b2-0-0-125",
  "status": "ACTIVE",
  "flavor": {
    "flavor_id": "bm1a-hf-1x7950x-128-2x960gb-2x10nic",
    "hardware_description": {
      "cpu": "1x AMD Ryzen 7950X 16C/32T 4.5 GHz",
      "disk": "2x 960GB NVMe"
    }
  },
  "addresses": {
    "pub_net": [
      {"addr": "203.0.113.10", "type": "fixed"}
    ]
  },
  "security_groups": [],
  "volumes": []
}
```

<Info>
Bare Metal servers have no security groups - `security_groups: []` - so all ports are accessible by default. To restrict traffic, configure OS-level firewall rules with `ufw` or `iptables` after connecting.
</Info>

<Info>
`volumes: []` is expected because Bare Metal servers use local NVMe storage - external block volumes cannot be attached and all data lives on the physical drives included with the server.
</Info>

<ApiH2>Step 6. Connect via SSH</ApiH2>

Use the public IP from Step 5.

```bash
ssh -i ~/.ssh/my-key ubuntu@203.0.113.10
```

The network interface inside the OS is `bond0`, not `eth0` - Bare Metal uses LACP bonding across physical ports:

```bash
$ ip a
5: bond0: ... inet 203.0.113.10/24 ... scope global bond0
6: bond0.2853@bond0: ... inet 10.0.0.125/24 ...
```

`bond0` is the public interface. `bond0.<vlan_id>` sub-interfaces appear if private networks were attached at creation.

<ApiH2>Clean up</ApiH2>

Bare Metal deletion is asynchronous and takes 3-5 minutes - the server must be fully deprovisioned before the hardware is released.

<Tabs>
  <Tab title="curl">
    ```bash
    curl -X DELETE \
      "https://api.gcore.com/cloud/v1/bminstances/$PROJECT_ID/$REGION_ID/$SERVER_ID" \
      -H "Authorization: APIKey $GCORE_API_KEY"
    ```
  </Tab>
  <Tab title="Python SDK">
    ```python
    client.cloud.baremetal.servers.delete(
        server_id=os.environ["SERVER_ID"],
        project_id=int(os.environ["PROJECT_ID"]),
        region_id=int(os.environ["REGION_ID"]),
    )
    ```
  </Tab>
</Tabs>

The API returns:

```json
{
  "tasks": ["127944f5-31c6-4438-9e9c-032f17109121"]
}
```

Poll <code>GET&nbsp;/cloud/v1/tasks/{task_id}</code> every 30 seconds until `state` is `FINISHED` - after deletion, <code>GET&nbsp;/cloud/v1/bminstances/{project_id}/{region_id}</code> returns `"count": 0`.

<Info>
Local NVMe storage is wiped when the server is deleted - there is no separate volume to delete, as all data on the physical drives is removed during deprovisioning.
</Info>
