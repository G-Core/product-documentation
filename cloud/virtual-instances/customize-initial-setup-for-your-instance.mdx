The **User Data** field on the Virtual Machine creation page accepts a [cloud-init](https://cloudinit.readthedocs.io/en/latest/) YAML document that runs when the VM boots for the first time. Use it to set passwords, create users, add SSH keys, install packages, write files, and configure network interfaces - without logging in to the VM afterward.

<MethodSwitch>
<MethodSection id="portal" label="Customer Portal">

By default, the **User Data** field is disabled for new Virtual Machines. Enable it by switching on the toggle and adding the cloud-init YAML content in the text box.

<Frame>![User Data content](/images/docs/cloud/virtual-instances/customize-initial-setup-for-your-instance/user-data-from-ui.png)</Frame>

</MethodSection>
<MethodSection id="api" label="REST API">

The `user_data` field in the instance creation request accepts a base64-encoded cloud-init YAML string. The encoding step is required - the API rejects plain-text YAML.

<Info>
A permanent [API token](/account-settings/api-tokens) is required, along with a project ID, region ID, image ID, and flavor ID. Retrieve them from the Customer Portal or by calling <code>GET&nbsp;/cloud/v1/projects</code>, <code>GET&nbsp;/cloud/v1/regions</code>, <code>GET&nbsp;/cloud/v1/images/{project_id}/{region_id}?visibility=public</code>, and <code>GET&nbsp;/cloud/v1/flavors/{project_id}/{region_id}</code>.
</Info>

Open a bash terminal and set these as environment variables before running the examples:

```bash
export GCORE_API_KEY="{YOUR_API_KEY}"
export PROJECT_ID="{YOUR_PROJECT_ID}"
export REGION_ID="{YOUR_REGION_ID}"
export IMAGE_ID="{YOUR_IMAGE_ID}"
export FLAVOR_ID="{YOUR_FLAVOR_ID}"
```

## Step 1. Write the cloud-init YAML

Create a YAML file with the cloud-init directives to run at boot. The file must start with the `#cloud-config` header on the first line.

The example below writes a file to `/tmp` to confirm initialization ran:

```yaml
#cloud-config
write_files:
  - path: /tmp/cloud-init-ok.txt
    content: |
      cloud-init ran successfully
    permissions: '0644'
```

Save this as `user-data.yaml`. The [cloud-init directives](#cloud-init-directives) section lists all available options.

## Step 2. Encode to base64

The `user_data` field requires the YAML encoded as a single-line base64 string.

```bash
USER_DATA_B64=$(base64 -w 0 user-data.yaml)
echo "$USER_DATA_B64"
```

The `-w 0` flag disables line-wrapping. On macOS, use `base64 -i user-data.yaml` instead.

## Step 3. Create the instance

Pass the encoded string in the `user_data` field of the [instance creation](https://api.gcore.com/docs/cloud#tag/Instances/operation/InstanceCreateSetV2.post) request.

| Parameter | Required | Description |
|-----------|----------|-------------|
| `user_data` | No | Base64-encoded cloud-init YAML. Runs on first boot only. |

<Tabs>
  <Tab title="curl">
    ```bash
    curl -X POST "https://api.gcore.com/cloud/v2/instances/$PROJECT_ID/$REGION_ID" \
      -H "Authorization: APIKey $GCORE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{
        "name_templates": ["lux-c2-{ip_octets}"],
        "flavor": "'"$FLAVOR_ID"'",
        "user_data": "'"$USER_DATA_B64"'",
        "interfaces": [{"type": "external"}],
        "volumes": [{
          "source": "image",
          "image_id": "'"$IMAGE_ID"'",
          "size": 20,
          "boot_index": 0
        }]
      }'
    ```
  </Tab>
  <Tab title="Python SDK">
    ```python
    import base64
    import os
    from gcore import Gcore

    client = Gcore(api_key=os.environ["GCORE_API_KEY"])

    with open("user-data.yaml", "rb") as f:
        user_data_b64 = base64.b64encode(f.read()).decode()

    instance = client.cloud.instances.create_and_poll(
        project_id=int(os.environ["PROJECT_ID"]),
        region_id=int(os.environ["REGION_ID"]),
        name_template="lux-c2-{ip_octets}",
        flavor=os.environ["FLAVOR_ID"],
        user_data=user_data_b64,
        interfaces=[{"type": "external"}],
        volumes=[{
            "source": "image",
            "image_id": os.environ["IMAGE_ID"],
            "size": 20,
            "boot_index": 0,
        }],
    )
    instance_id = instance.id
    print(f"Instance: {instance.id}  status: {instance.status}")
    ```
  </Tab>
</Tabs>

The API returns:

```json
{
  "tasks": ["b63f0fff-5d22-4c23-8e09-f7a3e1c1a999"]
}
```

For curl, poll <code>GET&nbsp;/cloud/v1/tasks/{task_id}</code> every 5 seconds until `state` is `FINISHED`, then read the instance ID from `created_resources.instances[0]`.

## Step 4. Verify cloud-init ran

After the instance is active, connect via SSH using the public IP from the Customer Portal or from the `addresses` field in the instance GET response, then check the sentinel file:

```bash
ssh ubuntu@{INSTANCE_IP} cat /tmp/cloud-init-ok.txt
```

The expected output is:

```text
cloud-init ran successfully
```

If the file is absent, check the [cloud-init logs](#check-cloud-init-logs) section for diagnostic commands.

## Terraform

The `gcore_instance` resource accepts the same base64-encoded string in the `user_data` attribute. Encode the YAML with the `base64` CLI tool first, then set the result directly in the configuration:

```hcl
resource "gcore_instance" "instance" {
  region_id  = var.region_id
  project_id = var.project_id
  name       = "my-instance"
  # ... other required fields ...
  user_data = "I2Nsb3VkLWNvbmZpZwp3cml0ZV9maWxlczo..."
}
```


## cloud-init directives

cloud-init is the standard initialization system for cloud Virtual Machines, supported by all major Linux distributions including Ubuntu, Debian, CentOS, Rocky Linux, AlmaLinux, Fedora, and others. It runs on the first boot, reads the `user_data` document, and executes the declared directives.

Every cloud-init document must begin with `#cloud-config` as its first line.

## Set a password for a virtual machine

By default, Gcore Virtual Machines allow SSH access with an SSH key only - password authentication is disabled. To enable password login, add `ssh_pwauth: True`:

```yaml
#cloud-config
password: your_password
chpasswd: { expire: False }
ssh_pwauth: True
power_state:
  mode: reboot
  timeout: 30
  condition: True
```

- `password` sets the value for the VM's default OS user. The default username depends on the image: `ubuntu` for Ubuntu, `centos` for CentOS, `debian` for Debian.
- `chpasswd: { expire: False }` skips the forced password change on first login.
- `ssh_pwauth: True` enables password-based SSH authentication.
- `power_state` instructs cloud-init to reboot the VM after initialization completes, which is required for the SSH configuration to take effect.

After the VM is running, connect with:

```bash
ssh {DEFAULT_USERNAME}@{INSTANCE_IP}
```

## Create a user

A new OS user can be created at boot using the `users` directive:

```yaml
#cloud-config
users:
  - name: guest
    sudo: ALL=(ALL) NOPASSWD:ALL
    passwd: {SHA-512_ENCRYPTED_VALUE}
    groups: users,admin
    lock_passwd: false
    shell: /bin/bash
power_state:
  mode: reboot
  timeout: 30
  condition: True
```

- `sudo` grants superuser rights. `ALL=(ALL) NOPASSWD:ALL` allows all commands without a password prompt.
- `passwd` takes a SHA-512-encrypted value. Generate it with `mkpasswd -m sha512crypt guest`.
- `groups` lists OS groups the user joins. Groups must exist before the user is added - see the [user groups](#configure-user-groups) section.
- `lock_passwd: false` allows password login for this user.
- `shell` sets the default shell for the user.

After the VM is ready, connect as the new user:

```bash
ssh guest@{INSTANCE_IP}
```

<Info>
To verify group membership after connecting, run `groups`. The output should include `users` and `admin`.
</Info>

## Enable root user

Root login is disabled by default. To enable root access at boot time, add:

```yaml
#cloud-config
disable_root: false
```

To enable root access on an already-running VM, follow the [root user](/cloud/virtual-instances/enable-root-user-on-a-linux-vm#enable-root-user-via-terminal) guide.

## Configure user groups

New OS groups can be created at boot using the `groups` directive:

```yaml
#cloud-config
groups:
  - regular-users
power_state:
  mode: reboot
  timeout: 30
  condition: True
```

After connecting to the VM, verify the group was created:

```bash
compgen -g
```

The `regular-users` group appears at the bottom of the output.

## Add an SSH key

Additional public SSH keys can be added to the VM at boot, granting access to other machines or team members:

```yaml
#cloud-config
ssh_authorized_keys:
  - ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABgQDLoSbg...
```

After the VM is running, verify the key was added:

```bash
cat ~/.ssh/authorized_keys
```

## Install packages

Packages available in the default OS repository need only to appear in the `packages` list. For packages requiring a third-party repository, declare the repository under `apt.sources` first:

```yaml
#cloud-config
apt:
  sources:
    docker.list:
      source: deb [arch=amd64] https://download.docker.com/linux/ubuntu $RELEASE stable
      keyid: 9DC858229FC7DD38854AE2D88D81803C0EBFCD88
package_update: true
package_upgrade: true
packages:
  - apt-transport-https
  - ca-certificates
  - gnupg
  - docker-ce
  - docker-ce-cli
  - nmap
  - traceroute
power_state:
  mode: reboot
  timeout: 30
  condition: True
```

`package_update` and `package_upgrade` refresh and upgrade the package index before installing. The `power_state` block restarts the VM after installation so the packages are available immediately on the path.

## Write files

Arbitrary files can be written at boot using the `write_files` directive:

```yaml
#cloud-config
write_files:
  - path: /root/setup.sh
    content: |
      #!/bin/bash
      echo "VM initialized"
    permissions: '0755'
```

After connecting, verify the file was created:

```bash
ls /root/setup.sh
cat /root/setup.sh
```

## Configure network interfaces

A static IP address can be added to a network interface by writing a second netplan configuration file at boot. When the VM starts, cloud-init writes `/etc/netplan/50-cloud-init.yaml` for the DHCP interface. A second file placed in `/etc/netplan` is merged with it at reboot, applying the static address:

```yaml
#cloud-config
write_files:
  - content: |
      network:
        version: 2
        renderer: networkd
        ethernets:
          enp3s0:
            addresses:
              - 192.170.1.25/24
              - 2020:1::1/64
            nameservers:
              addresses:
                - 8.8.8.8
                - 8.8.4.4
    path: /etc/netplan/00-add-static-ip.yaml
    permissions: '0644'
power_state:
  mode: reboot
  timeout: 30
  condition: True
```

After the VM reboots, verify the static address was applied:

```bash
ip a
```

The `enp3s0` interface should show both the static IPv4 address `192.170.1.25` and IPv6 address `2020:1::1` alongside the DHCP-assigned address.

## Check cloud-init logs

When cloud-init directives do not apply as expected, two log files inside the VM contain diagnostic information.

### cloud-init log

`/var/log/cloud-init.log` records the full execution trace, including errors:

```bash
cat /var/log/cloud-init.log
```

### cloud-init output log

`/var/log/cloud-init-output.log` captures stdout and stderr from the scripts cloud-init ran:

```bash
cat /var/log/cloud-init-output.log
```
