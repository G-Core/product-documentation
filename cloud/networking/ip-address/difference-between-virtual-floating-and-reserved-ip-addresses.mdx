Cloud IP addresses serve different purposes based on their type. Virtual, floating, and reserved IPs provide flexibility for high availability, external access, and network management. The table below outlines their key differences and use cases.

| IP Type | Use cases | Intended for | Subnet | Failover | Requirements |
| --- | --- | --- | --- | --- | --- |
| Virtual IP | - Assign to multiple Virtual Machines (VMs)<br />- Use as a secondary IP<br />- Used by Load Balancers (LBs)<br />- Subtype of Reserved IP | VMs | Public<br />or private | Yes | Multiple VMs |
| Floating IP | - Public access via DNAT<br />- Assign to VM, Bare Metal,<br />or LB | Private ports of VMs,<br />Bare Metal, and LBs | Public<br />(via DNAT) | Yes | Private network |
| Reserved IP | - Pre-allocate IP<br />- Assign anytime<br />- Convert to Virtual IP<br />- Bare Metal: contact [support](https://support.gcore.com/) | VMs, Bare Metal,<br />and LBs | Public<br />or private | No | None |

## Decision guide

If you need a public IP for your VM:

- If your VM has a public interface, the platform automatically assigns a public IP address (one per VM).
- If your VM runs in a private subnet, use a floating IP to enable external access.

If you need multiple IPs:

- For VMs, you can use one public IP per VM and attach additional floating IPs if needed.
- For Bare Metal servers, contact [support](https://support.gcore.com/) to request additional IPs.

## Cloud IPs

All IP addresses on the Cloud platform are native and correspond to the exact physical location where the server is deployed. This results in low latency, optimized routing, and improved performance for cloud services.

When a public IP is assigned, it comes from a shared regional pool and remains static for the lifetime of the network interface. However, if the interface is deleted and re-created, the public IP is released back into the pool and replaced with a new one.
