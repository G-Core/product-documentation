<p>A Load Balancer distributes incoming requests across Virtual Machines to improve infrastructure fault tolerance.</p>

## Step 1. Initiate Load Balancer creation

<p>In the project, navigate to **Load Balancers** in the **Networking** section and click **Create Load Balancer**.</p>

<Frame>
  ![Load Balancers list with Create Load Balancer button](/images/docs/cloud/networking/create-and-configure-a-load-balancer/lb-create-button.png)
</Frame>

<p>The new page opens. Select a region for balancing.</p>

<Warning>
  Traffic can only be balanced within a single data center.
</Warning>

## Step 2. Set computing configuration

<p>Select a flavor for the Load Balancer — vCPU and RAM. All Load Balancers are created in high availability mode with active-standby instances. Upon failure of the active instance, the standby one takes over the load-balancing functions.</p>

<Frame>
  ![Load Balancer type selection with Standard flavor options](/images/docs/cloud/networking/create-and-configure-a-load-balancer/lb-flavor-type.png)
</Frame>

<p>Two classes of Load Balancer flavors are available:</p>

- **Standard** — general-purpose configurations suitable for most workloads, with network throughput up to 1.5 Gbps.
- **Network-Optimized** — built on network-optimized infrastructure with SR-IOV on the VIP port. These deliver higher packet rates and lower latency, with guaranteed bandwidth up to 10 Gbps per instance.

<Frame>
  ![Network-Optimized flavor tab](/images/docs/cloud/networking/create-and-configure-a-load-balancer/lb-type-network-optimized.png)
</Frame>

| **Class** | **vCPU** | **RAM (GiB)** | **Network bandwidth (VIP)** |
| --- | --- | --- | --- |
| Standard | 1 | 2 | 1.5 Gbps |
| Standard | 2 | 4 | 5 Gbps |
| Network-Optimized | 4 | 8 | 10 Gbps |
| Network-Optimized | 8 | 16 | 10 Gbps |

<p>Use Network-Optimized flavors for services that handle large volumes of concurrent connections or high-bandwidth traffic — media streaming, CDN origin servers, and API gateways.</p>

## Step 4. Configure preferred connectivity

<p>L2 (Layer 2) and L3 (Layer 3) connectivity options are available. This setting determines the preferred connectivity method the Load Balancer uses to connect to backend pool members.</p>

<p>The Load Balancer determines available routes by evaluating subnet host routes only. Due to current system limitations, it does not take router `host_routes` into account.</p>

<p>If the preferred connectivity is not feasible, traffic will automatically route via the alternative method:</p>

**L2 (preferred) -> (if not possible) L3 -> (if not possible) Validation Error**

**L3 (preferred) -> (if not possible) L2 -> (if not possible) Validation Error**

<Tabs>
  <Tab title="L2 (Layer 2)">
    <p>L2 connectivity offers better performance because traffic flows directly between the Load Balancer and pool members without passing through a router. This reduces network hops and minimizes latency.</p>

    <p>However, this approach requires more IP addresses. In networks with many `/24` subnets, each Load Balancer must create ports in every subnet where its members are located. This can lead to high IP utilization and reduced efficiency in large-scale deployments.</p>

    <Frame>
      ![Select Layer 2 connectivity option](/images/docs/cloud/networking/create-and-configure-a-load-balancer/step-5-preferred-connectivity.png)
    </Frame>
  </Tab>
  <Tab title="L3 (Layer 3)">
    <p>L3 connectivity routes traffic through a router or gateway, introducing additional network hops that may slightly impact performance.</p>

    <p>It also optimizes IP address utilization by reducing the number of required IPs per Load Balancer. Instead of allocating a separate IP in every subnet, the Load Balancer communicates with pool members across subnets using routing mechanisms. This approach improves scalability and efficiency, especially in environments with multiple subnets.</p>

    <Tip>
      For a single subnet setup, use **L2 connectivity** for optimal performance. For deployments with multiple subnets or complex networking requirements, contact support to determine the best configuration.
    </Tip>
  </Tab>
</Tabs>

## Step 5. Configure listeners

<p>Add listeners that will check for connection requests using the protocol and port specified. Multiple listeners are supported per Load Balancer.</p>

<Steps>
  <Step title="Add a listener">
    In the **Listeners** section, click **Add Listener**.

    <Frame>
      ![Listeners section with Add Listener button, and Load Balancer Name, Logging, and Tags sections below](/images/docs/cloud/networking/create-and-configure-a-load-balancer/lb-listeners-name-logging.png)
    </Frame>
  </Step>
  <Step title="Enter the listener name">
    Enter the listener's name.
  </Step>
  <Step title="Select the protocol">
    Select the required protocol: TCP, UDP, HTTP, HTTPS, Terminated HTTPS, or Prometheus. Multiple TLS certificates are configurable for Terminated HTTPS and Prometheus.
  </Step>
  <Step title="Specify the port">
    Specify a port the Load Balancer will listen on for incoming traffic. Keep the default port for protocols that provide one, or enter a custom port from 1 to 65535.
  </Step>
  <Step title="(Optional) Enable X-Forwarded headers">
    To identify the origin of the client IP address when a request passes through the load balancer, enable the **Add headers X-Forwarded-For, X-Forwarded-Port, X-Forwarded-Proto to requests** toggle.
  </Step>
  <Step title="(Optional) Configure TLS certificates">
    For Terminated HTTPS and Prometheus protocols, [TLS certificates](/cloud/networking/load-balancers/add-certificates-to-load-balancer) are configurable.
  </Step>
  <Step title="Set the connection limit">
    Set the connection limit — the maximum number of simultaneous connections handled by this listener. The default value is 100,000. Setting the value to -1 sets the limit to the default.
  </Step>
  <Step title="(Optional) Add allowed CIDR ranges">
    Add allowed CIDR ranges to define which IP addresses can access the content. All IP addresses outside the specified ranges will be denied access. Up to 100 CIDR ranges can be specified per listener.

    The IP version of Allowed CIDRs must match the IP version of the Load Balancer's Virtual IPs (VIPs):

    - VIPs with both IPv4 and IPv6 support CIDRs of both versions.
    - VIPs with only IPv4 allow only IPv4 CIDRs.
    - VIPs with only IPv6 allow only IPv6 CIDRs.

    If the provided CIDRs do not match the VIP's IP version, the API will return a validation error.

    <Frame>
      ![Listener form showing Allowed CIDRs field with an IPv4 example](/images/docs/cloud/networking/create-and-configure-a-load-balancer/step-8-edit-listener.png)
    </Frame>
  </Step>
  <Step title="(Optional) Configure user authentication">
    For HTTP-based listeners, configure basic user authentication to protect the resource from unauthorized access. Click **+ Add user** to enable authentication:

    - **Enter username**: specify a username.
    - **Password**: specify a password or choose the **Encrypted password** option to store the password as a hash — [encrypted passwords](/cloud/networking/load-balancers/add-certificates-to-load-balancer#create-an-encrypted-password) are stored as bcrypt hashes.

    <Info>
      A password must contain at least one lowercase character, one uppercase character, at least one number, and a special character. The maximum password length is 128 symbols.
    </Info>
  </Step>
  <Step title="Create the listener">
    Click **Create Listener**.

    <Frame>
      ![Add Listener dialog with name, protocol, port, connection limit, and allowed CIDRs fields](/images/docs/cloud/networking/create-and-configure-a-load-balancer/lb-listener-dialog.png)
    </Frame>
  </Step>
</Steps>

<p>Listener configuration options differ depending on the selected protocol. HTTP and Terminated HTTPS protocols allow additional settings — headers and authentication — while TCP and UDP listeners focus on specifying ports and connection limits.</p>

### Pool

<p>Configure a pool — a list of VMs to which the listener will redirect incoming traffic. Click **Add new pool** in the **Listeners** section to start configuring.</p>

<Frame>
  ![Add pools](/images/docs/cloud/networking/create-and-configure-a-load-balancer/step-4-pools-lb.png)
</Frame>

<Steps>
  <Step title="Name the pool">
    Specify the pool name.
  </Step>
  <Step title="Select the balancing algorithm">
    Select one of the following:

    - **Round robin** — requests are distributed across servers within a cluster one by one: the first request is sent to the first server, the second to the second, and so on in a circle.
    - **Least Connection** — new requests are sent to a server with the fewest active connections.
    - **Source IP** — a client's IP address is used to determine which server receives the request.
  </Step>
  <Step title="Select the pool protocol">
    Select the protocol the Load Balancer uses to communicate with backend servers. The available options are: **HTTP**, **HTTPS**, **TCP**, **PROXY**, and **PROXY V2**. The pool protocol is independent from the listener protocol and should match what the backend application expects.

    <Frame>
      ![Pool protocol dropdown showing HTTP, HTTPS, TCP, UDP, PROXY, and PROXY V2 options](/images/docs/cloud/networking/create-and-configure-a-load-balancer/pool-protocol-dropdown.png)
    </Frame>

    To preserve the original client IP address on the backend, select **PROXY** (Proxy Protocol v1, text-based) or **PROXY V2** (Proxy Protocol v2, binary). When either option is selected, the Load Balancer prepends a Proxy Protocol header to each TCP connection, allowing backend servers to read the real client IP instead of the Load Balancer's IP.

    <Info>
      PROXY and PROXY V2 require the listener to use the TCP protocol. The backend application must be configured to parse Proxy Protocol headers. Selecting PROXY or PROXY V2 on a pool whose members do not expect these headers will cause connection failures.
    </Info>
  </Step>
  <Step title="(Optional) Configure sticky sessions">
    In the **Sticky Sessions** dropdown, select the session persistence method:

    - **None** — no session persistence, requests are distributed according to the selected algorithm.
    - **App Cookie** — the Load Balancer reads an existing cookie set by the backend application and uses it to route subsequent requests from the same client to the same server. Fill in the **Cookie** field with the cookie name.
    - **HTTP Cookie** — the Load Balancer generates and manages its own session cookie automatically.

    <Frame>
      ![Pool configuration showing name, algorithm, protocol, and sticky sessions fields](/images/docs/cloud/networking/create-and-configure-a-load-balancer/step-4-pool-conf-lb.png)
    </Frame>
  </Step>
</Steps>

**Sharing a pool across multiple listeners**

<p>A single pool can be used by multiple listeners within the same Load Balancer. This reduces configuration overhead and simplifies pool member management when routing traffic from different ports to the same backend servers.</p>

<Info>
  Shared pools can only be configured via the [Load Balancer API](https://apidocs.gcore.com/cloud#tag/Load-Balancers).
</Info>

<p>To configure a shared pool:</p>

1. Create a Load Balancer via `POST /cloud/v1/loadbalancers/{project_id}/{region_id}`.
2. Create a pool via `POST /cloud/v1/lbpools/{project_id}/{region_id}`, specifying `loadbalancer_id` without `listener_id`.
3. Create listeners via `POST /cloud/v1/lblisteners/{project_id}/{region_id}`, referencing the pool via `default_pool_id`.

<Warning>
  When deleting a listener that uses a shared pool, do not use the `delete_default_pool` option. This would delete the pool and affect all other listeners using it. Instead, manage the pool separately.
</Warning>

<p>A [load balancer](/api-reference/cloud/load-balancers/create-load-balancer) distributes incoming requests across backend VMs, routing traffic only to healthy members. <Info>

</Info>

<p>Open a bash terminal and set these as environment variables before running the examples:</p>

```bash
export GCORE_API_KEY="{YOUR_API_KEY}"

export GCORE_CLOUD_REGION_ID="{YOUR_REGION_ID}"
export VM1_IP="{VM1_PUBLIC_IP}"
export VM2_IP="{VM2_PUBLIC_IP}"
```

<Info>
**VM1\_IP** and **VM2\_IP** are the public IP addresses of the virtual machines that will serve as backend pool members. To create VMs and find their IPs, see [Create a virtual machine](/cloud/virtual-instances/create-an-instance).
</Info>

## Quickstart

<p>The scripts below create a load balancer with an HTTP listener and a round-robin pool, then add two backend VMs as pool members. Set `VM1_IP` and `VM2_IP` to the public IP addresses of running VMs.</p>

<Tabs>
  <Tab title="Python SDK">
    ```python
    import os
    from gcore import Gcore

    client = Gcore()

    # Step 1. List flavors and pick the smallest
    flavors = client.cloud.load_balancers.flavors.list()
    lb_flavor = flavors.results[0].flavor_name
    print(f"Flavor: {lb_flavor}")

    # Step 2. Create load balancer (takes about 1 minute)
    lb = client.cloud.load_balancers.create_and_poll(
        name="my-lb",
        flavor=lb_flavor,
        polling_timeout_seconds=300,
    )
    print(f"LB VIP: {lb.vip_address}")

    # Step 3. Create HTTP listener on port 80
    listener = client.cloud.load_balancers.listeners.create_and_poll(
        load_balancer_id=lb.id,
        name="http-listener",
        protocol="HTTP",
        protocol_port=80,
    )

    # Step 4. Create round-robin pool
    pool = client.cloud.load_balancers.pools.create_and_poll(
        name="http-pool",
        lb_algorithm="ROUND_ROBIN",
        protocol="HTTP",
        listener_id=listener.id,
        load_balancer_id=lb.id,
    )

    # Step 5. Add backend VMs as pool members
    for ip in [os.environ["VM1_IP"], os.environ["VM2_IP"]]:
        t = client.cloud.load_balancers.pools.members.create(
            pool_id=pool.id,
            address=ip,
            protocol_port=80,
        )
        client.cloud.tasks.poll(t.tasks[0], polling_interval_seconds=5, polling_timeout_seconds=120)

    print(f"Traffic on {lb.vip_address}:80 is now distributed across both VMs")
    ```
  </Tab>
  <Tab title="Go SDK">
    ```go
    package main

    import (
        "context"
        "fmt"
        "os"

        gcore "github.com/G-Core/gcore-go"
        "github.com/G-Core/gcore-go/cloud"
    )

    func main() {
        client := gcore.NewClient()
        ctx := context.TODO()

        // Step 1. List flavors and pick the smallest
        flavors, err := client.Cloud.LoadBalancers.Flavors.List(ctx, cloud.LoadBalancerFlavorListParams{})
        if err != nil {
            panic(err.Error())
        }
        lbFlavor := flavors.Results[0].FlavorName
        fmt.Printf("Flavor: %s\n", lbFlavor)

        // Step 2. Create load balancer (takes about 1 minute)
        lb, err := client.Cloud.LoadBalancers.NewAndPoll(ctx, cloud.LoadBalancerNewParams{
            Name:   "my-lb",
            Flavor:    gcore.String(lbFlavor),
        })
        if err != nil {
            panic(err.Error())
        }
        fmt.Printf("LB VIP: %s\n", lb.VipAddress)

        // Step 3. Create HTTP listener on port 80
        listener, err := client.Cloud.LoadBalancers.Listeners.NewAndPoll(ctx, cloud.LoadBalancerListenerNewParams{
            LoadBalancerID: lb.ID,
            Name:           "http-listener",
            Protocol:       "HTTP",
            ProtocolPort:   80,
        })
        if err != nil {
            panic(err.Error())
        }

        // Step 4. Create round-robin pool
        pool, err := client.Cloud.LoadBalancers.Pools.NewAndPoll(ctx, cloud.LoadBalancerPoolNewParams{
            Name:           "http-pool",
            LbAlgorithm:    "ROUND_ROBIN",
            Protocol:       "HTTP",
            ListenerID:     gcore.String(listener.ID),
            LoadBalancerID: gcore.String(lb.ID),
        })
        if err != nil {
            panic(err.Error())
        }

        // Step 5. Add backend VMs as pool members
        for _, ip := range []string{os.Getenv("VM1_IP"), os.Getenv("VM2_IP")} {
            t, err := client.Cloud.LoadBalancers.Pools.Members.New(ctx, pool.ID, cloud.LoadBalancerPoolMemberNewParams{
                Address:      ip,
                ProtocolPort: 80,
            })
            if err != nil {
                panic(err.Error())
            }
            if _, err = client.Cloud.Tasks.Poll(ctx, t.Tasks[0]); err != nil {
                panic(err.Error())
            }
        }
        fmt.Printf("Traffic on %s:80 is now distributed across both VMs\n", lb.VipAddress)
    }
    ```
  </Tab>
</Tabs>

### Step 2. Create the load balancer

<p>Creating the load balancer without specifying a network assigns a public VIP automatically. Provisioning takes about 1 minute.</p>

| `flavor` | Yes | The `flavor_name` string from step 1, not the `flavor_id` UUID. |

### Step 4. Create a pool

<p>A [pool](/api-reference/cloud/load-balancers/create-load-balancer-pool) groups the backend VMs and distributes requests across them according to the chosen algorithm.</p>

| `load_balancer_id` | Yes | The LB ID from step 2. |
| `lb_algorithm` | Yes | `ROUND_ROBIN`, `LEAST_CONNECTIONS`, or `SOURCE_IP`. |
| `protocol` | Yes | Must match the listener protocol (`HTTP` in this example). |

<Tabs>

## Quickstart

<p>Deploys an HTTP load balancer with a round-robin pool and two backend members.</p>

```hcl
resource "gcore_cloud_load_balancer" "example" {
  project_id = var.project_id
  region_id  = var.region_id
  name       = "my-lb"
  flavor     = "lb1-1-2"
}

resource "gcore_cloud_load_balancer_listener" "http" {
  project_id       = var.project_id
  region_id        = var.region_id
  load_balancer_id = gcore_cloud_load_balancer.example.id
  name             = "http-listener"
  protocol         = "HTTP"
  protocol_port    = 80
}

resource "gcore_cloud_load_balancer_pool" "backend" {
  project_id       = var.project_id
  region_id        = var.region_id
  load_balancer_id = gcore_cloud_load_balancer.example.id
  listener_id      = gcore_cloud_load_balancer_listener.http.id
  name             = "backend-pool"
  lb_algorithm     = "ROUND_ROBIN"
  protocol         = "HTTP"

  members = [
    {
      address       = "192.0.2.10"  # VM1's IP address
      protocol_port = 80
    },
    {
      address       = "192.0.2.11"  # VM2's IP address
      protocol_port = 80
    }
  ]
}

output "vip_address" {
  value = gcore_cloud_load_balancer.example.vip_address
}
```

<p>Run `terraform init && terraform apply`. Load balancer creation takes approximately one minute.</p>

## Step-by-step

<p>Each resource is covered independently below with its key attributes and options.</p>

<Accordion title="Show all steps">

### Step 1. Create the load balancer

<p>The `gcore_cloud_load_balancer` resource provisions the balancer and its virtual IP (VIP).</p>

| Attribute | Required | Description |
|-----------|----------|-------------|
| `name` | Yes | Display name of the load balancer. |
| `flavor` | No | Compute flavor. Omit to use the region default. |
| `vip_network_id` | No | Network ID for the VIP. Omit for a public VIP. |
| `vip_subnet_id` | No | Subnet ID for the VIP within the selected network. |
| `vip_ip_family` | No | IP version when auto-selecting a subnet. Values: `ipv4` (default), `ipv6`, `dual`. |
| `preferred_connectivity` | No | How backends connect to the VIP. Values: `L2` (better performance), `L3` (fewer IPs used). |

<p>Available flavors and their vCPU/RAM configurations:</p>

| Flavor | vCPU | RAM | Max bandwidth |
|--------|------|-----|---------------|
| `lb1-1-2` | 1 | 2 GiB | 1.5 Gbps |
| `lb1-2-4` | 2 | 4 GiB | 5 Gbps |
| `lb1-4-8` | 4 | 8 GiB | 10 Gbps |
| `lb1-8-16` | 8 | 16 GiB | 10 Gbps |

<p>To attach a public floating IP to the load balancer's VIP:</p>

```hcl
resource "gcore_cloud_load_balancer" "example" {
  project_id = var.project_id
  region_id  = var.region_id
  name       = "my-lb"
  flavor     = "lb1-1-2"

  floating_ip = {
    source = "new"  # use "existing" and set existing_floating_id to reuse one
  }
}
```

### Step 2. Create a listener

<p>A listener defines the protocol and port on which the load balancer accepts traffic. One load balancer supports multiple listeners.</p>

| Attribute | Required | Description |
|-----------|----------|-------------|
| `load_balancer_id` | Yes | ID of the parent load balancer. |
| `name` | Yes | Display name of the listener. |
| `protocol` | Yes | Traffic protocol. Values: `HTTP`, `HTTPS`, `TCP`, `UDP`, `TERMINATED_HTTPS`. |
| `protocol_port` | Yes | Port number the listener binds to. |
| `secret_id` | Required for `TERMINATED_HTTPS` | ID of the default TLS certificate from Secrets Manager. Used as fallback when no SNI match is found. |
| `sni_secret_id` | No | List of additional certificate IDs for SNI-based domain matching. |

<p>For `TERMINATED_HTTPS`, set `secret_id` to the ID of the default TLS certificate. Add `sni_secret_id` to serve different certificates for different domains — the default certificate from `secret_id` is used when no SNI match is found:</p>

```hcl
resource "gcore_cloud_load_balancer_listener" "https" {
  project_id       = var.project_id
  region_id        = var.region_id
  load_balancer_id = gcore_cloud_load_balancer.example.id
  name             = "https-listener"
  protocol         = "TERMINATED_HTTPS"
  protocol_port    = 443
  secret_id        = var.tls_secret_id    # ID of a gcore_cloud_secret resource
  # sni_secret_id = [var.sni_cert_id]     # optional: additional certs for multi-domain
}
```

<p>For an HTTP listener, omit `secret_id` and `sni_secret_id`:</p>

```hcl
resource "gcore_cloud_load_balancer_listener" "http" {
  project_id       = var.project_id
  region_id        = var.region_id
  load_balancer_id = gcore_cloud_load_balancer.example.id
  name             = "http-listener"
  protocol         = "HTTP"
  protocol_port    = 80
}
```

### Step 3. Create a pool with backend members

<p>A pool groups backend VMs and defines how traffic is distributed among them. Members are declared inline using the `members` attribute.</p>

| Attribute | Required | Description |
|-----------|----------|-------------|
| `load_balancer_id` | Yes | ID of the parent load balancer. |
| `listener_id` | No | ID of the listener to attach the pool to. |
| `name` | Yes | Display name of the pool. |
| `lb_algorithm` | Yes | Distribution algorithm. Values: `ROUND_ROBIN`, `LEAST_CONNECTIONS`, `SOURCE_IP`. |
| `protocol` | Yes | Backend protocol. Must match the listener protocol (use `HTTP` for both `HTTP` and `TERMINATED_HTTPS` listeners). |

<p>Each entry in the `members` list supports the following fields:</p>

| Field | Required | Description |
|-------|----------|-------------|
| `address` | Yes | IP address of the backend VM. |
| `protocol_port` | Yes | Port the application listens on inside the VM. |
| `weight` | No | Relative weight for traffic distribution. Range: 1–256, default 1. |
| `backup` | No | Set to `true` to mark as a backup member. Traffic goes to backup members only when all primary members are unreachable. |

<p>Example pool with weighted members:</p>

```hcl
resource "gcore_cloud_load_balancer_pool" "backend" {
  project_id       = var.project_id
  region_id        = var.region_id
  load_balancer_id = gcore_cloud_load_balancer.example.id
  listener_id      = gcore_cloud_load_balancer_listener.http.id
  name             = "backend-pool"
  lb_algorithm     = "ROUND_ROBIN"
  protocol         = "HTTP"

  members = [
    {
      address       = "192.0.2.10"
      protocol_port = 80
      weight        = 3
    },
    {
      address       = "192.0.2.11"
      protocol_port = 80
      weight        = 1
    }
  ]
}
```

### Step 4. (Optional) Add a health monitor

<p>A health monitor probes each pool member on a schedule and removes unreachable members from rotation automatically. Define it inline in the pool resource using the `healthmonitor` attribute.</p>

| Field | Required | Description |
|-------|----------|-------------|
| `type` | Yes | Probe protocol. Values: `HTTP`, `HTTPS`, `TCP`, `PING`, `UDP-CONNECT`, `TLS-HELLO`. |
| `delay` | Yes | Seconds between probes. |
| `timeout` | Yes | Seconds to wait for a probe response. Must be less than `delay`. |
| `max_retries` | Yes | Consecutive successes before marking a member `ONLINE`. |
| `url_path` | No | HTTP path to probe. Default is `/`. Only for `HTTP` and `HTTPS` types. |
| `expected_codes` | No | Expected HTTP response codes, e.g. `200` or `200,204`. Default is `200`. |

<p>Example pool block with a health monitor:</p>

```hcl
resource "gcore_cloud_load_balancer_pool" "backend" {
  # ... other attributes ...

  healthmonitor = {
    type        = "HTTP"
    delay       = 10
    timeout     = 5
    max_retries = 3
    url_path    = "/health"
  }
}
```
