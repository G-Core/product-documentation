Gcore Load Balancers support mutual TLS (mTLS) authentication for two-way certificate verification between clients and servers.

mTLS can be configured at two levels:

- **Client to Load Balancer**: Validates client certificates at the listener level
- **Load Balancer to backend servers**: Validates server certificates at the pool level

## How it works

When mTLS is enabled, both parties exchange and verify certificates before establishing a connection. The Load Balancer uses secrets stored in Gcore Secrets Manager to manage certificates.

| Parameter | Level | Purpose |
|-----------|-------|---------|
| `secret_id` | Listener | Server TLS certificate for HTTPS termination |
| `ca_secret_id` | Pool | CA certificate to validate backend server certificates |
| `secret_id` | Pool | Client certificate for the Load Balancer to authenticate to backends |

## Prerequisites

- Active Gcore account with Load Balancer access
- TLS certificates in PKCS12 or PEM format
- CA certificates for backend validation (if configuring backend mTLS)

## Configure client to Load Balancer mTLS

### 1\. Add server certificate to Secrets Manager

Create a secret containing the server certificate:

```
POST /v1/secrets/{project_id}/{region_id}
```

```json
{
  "name": "lb-server-cert",
  "secret_type": "certificate",
  "payload": "<base64-encoded-pkcs12-certificate>",
  "payload_content_type": "application/octet-stream",
  "payload_content_encoding": "base64"
}
```

<Warning>
Using `payload_content_type: "text/plain"` with base64 encoding causes secret creation to fail. Use `application/octet-stream` instead.
</Warning>

### 2\. Create Load Balancer with mTLS listener

Create a Load Balancer with a `TERMINATED_HTTPS` listener that references the server certificate:

```
POST /v1/loadbalancers/{project_id}/{region_id}
```

```json
{
  "name": "mtls-lb",
  "flavor": "lb1-1-2",
  "vip_network_id": "<network-id>",
  "listeners": [
    {
      "name": "mtls-listener",
      "protocol": "TERMINATED_HTTPS",
      "protocol_port": 443,
      "secret_id": "<server-certificate-secret-id>",
      "pools": [
        {
          "name": "backend-pool",
          "protocol": "HTTP",
          "lb_algorithm": "ROUND_ROBIN",
          "members": [
            {
              "address": "192.168.1.10",
              "protocol_port": 80
            }
          ]
        }
      ]
    }
  ]
}
```

## Configure Load Balancer to backend mTLS

### 1\. Add CA certificate to Secrets Manager

Create a secret containing the CA certificate that validates backend server certificates:

```
POST /v1/secrets/{project_id}/{region_id}
```

```json
{
  "name": "backend-ca-cert",
  "secret_type": "certificate",
  "payload": "<base64-encoded-ca-certificate>",
  "payload_content_type": "application/octet-stream",
  "payload_content_encoding": "base64"
}
```

### 2\. (Optional) Add Load Balancer client certificate

If backend servers require client authentication, create a secret with the Load Balancer client certificate:

```json
{
  "name": "lb-client-cert",
  "secret_type": "certificate",
  "payload": "<base64-encoded-client-certificate>",
  "payload_content_type": "application/octet-stream",
  "payload_content_encoding": "base64"
}
```

### 3\. Create pool with mTLS configuration

Create a pool with `HTTPS` protocol and reference the CA certificate:

```
POST /v1/loadbalancers/{project_id}/{region_id}/{loadbalancer_id}/pools
```

```json
{
  "name": "secure-backend-pool",
  "protocol": "HTTPS",
  "lb_algorithm": "ROUND_ROBIN",
  "ca_secret_id": "<ca-certificate-secret-id>",
  "secret_id": "<lb-client-certificate-secret-id>",
  "members": [
    {
      "address": "192.168.1.20",
      "protocol_port": 443
    }
  ]
}
```

<Info>
The `ca_secret_id` parameter enables the Load Balancer to validate backend server certificates. The `secret_id` parameter provides the client certificate for mutual authentication.
</Info>

## Add mTLS to existing Load Balancer

### 1\. Create required secrets

Follow the steps above to create certificate secrets in Secrets Manager.

### 2\. Update listener

```
PUT /v1/lblisteners/{project_id}/{region_id}/{listener_id}
```

```json
{
  "secret_id": "<server-certificate-secret-id>"
}
```

### 3\. Update pool

```
PUT /v1/lbpools/{project_id}/{region_id}/{pool_id}
```

```json
{
  "protocol": "HTTPS",
  "ca_secret_id": "<ca-certificate-secret-id>",
  "secret_id": "<lb-client-certificate-secret-id>"
}
```

## Certificate format requirements

| Certificate type | Format | Notes |
|------------------|--------|-------|
| Listener server certificate | PKCS12 | Contains certificate, private key, and intermediate chain |
| CA certificate | PEM, base64 encoded | Used to validate backend servers |
| Client certificate | PKCS12 or PEM | Used for Load Balancer to authenticate to backends |

<Info>
Each API call returns a `task_id`. Wait for task completion before proceeding to the next step.
</Info>
