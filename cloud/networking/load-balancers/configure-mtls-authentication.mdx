Gcore Load Balancers support mutual TLS (mTLS) authentication that enables two-way authentication between the client and the server. This feature ensures that traffic is encrypted and both parties are verified before any data exchange.

mTLS authentication can be configured at two levels:

- **Client ↔ Load Balancer (Listener level)**: Validates client certificates connecting to the load balancer
- **Load Balancer ↔ Backend Servers (Pool level)**: Validates server certificates when the load balancer connects to backend members

## Key mTLS parameters

| Parameter | Location | Purpose |
|-----------|----------|---------|
| `listener.secret_id` | Listener | Server TLS certificate for HTTPS termination |
| `pool.ca_secret_id` | Pool | CA certificate to validate backend server certificates (LB → server mTLS) |
| `pool.secret_id` | Pool | Client certificate for LB to authenticate to backend servers |
| `pool.crl_secret_id` | Pool | Certificate Revocation List (optional) |

## Secret creation

All secrets must be created using `POST /v1/secrets/{project_id}/{region_id}` with the following format:

```json
{
  "name": "certificate-name",
  "secret_type": "certificate",
  "payload": "<base64-encoded-certificate>",
  "payload_content_type": "application/octet-stream",
  "payload_content_encoding": "base64"
}
```

<Warning>
Using `payload_content_type: "text/plain"` with `payload_content_encoding: "base64"` will cause secret creation to fail.
</Warning>

### Valid parameter combinations

| payload_content_encoding | Valid payload_content_type values |
|--------------------------|-----------------------------------|
| `"base64"` | `"application/octet-stream"`, `"application/pkcs8"`, `null` |

## Configuration examples

The following examples demonstrate complete workflows for configuring mTLS authentication at different levels.

### Example 1: Client ↔ Load Balancer mTLS

This configuration validates client certificates connecting to the load balancer.

**Step 1: Create server certificate secret (for HTTPS termination)**

```
POST /v1/secrets/{project_id}/{region_id}
Content-Type: application/json
```

```json
{
  "name": "lb-server-cert",
  "secret_type": "certificate",
  "payload": "<base64-encoded-pkcs12-certificate>",
  "payload_content_type": "application/octet-stream",
  "payload_content_encoding": "base64"
}
```

**Response:**

```json
{"task_id": "task-001"}
```

Wait for task completion, then retrieve the `secret_id` from the task result.

**Step 2: Create Load Balancer with mTLS listener**

```
POST /v1/loadbalancers/{project_id}/{region_id}
Content-Type: application/json
```

```json
{
  "name": "client-mtls-lb",
  "flavor": "lb1-1-2",
  "vip_network_id": "<network-id>",
  "listeners": [
    {
      "name": "client-mtls-listener",
      "protocol": "TERMINATED_HTTPS",
      "protocol_port": 443,
      "secret_id": "<lb-server-cert-id>",
      "insert_x_forwarded": true,
      "pools": [
        {
          "name": "backend-pool",
          "protocol": "HTTP",
          "lb_algorithm": "ROUND_ROBIN",
          "members": [
            {
              "address": "192.168.1.10",
              "protocol_port": 80,
              "weight": 1
            }
          ],
          "healthmonitor": {
            "type": "HTTP",
            "delay": 10,
            "timeout": 5,
            "max_retries": 3,
            "url_path": "/health"
          }
        }
      ]
    }
  ]
}
```

**Traffic flow:** Client (with certificate) → mTLS → Load Balancer → HTTP → Backend Servers

### Example 2: Load Balancer ↔ Backend Servers mTLS

This configuration validates backend server certificates when the load balancer connects to them.

**Step 1: Create CA certificate secret (to validate backend servers)**

```
POST /v1/secrets/{project_id}/{region_id}
Content-Type: application/json
```

```json
{
  "name": "backend-ca-cert",
  "secret_type": "certificate",
  "payload": "<base64-encoded-ca-certificate>",
  "payload_content_type": "application/octet-stream",
  "payload_content_encoding": "base64"
}
```

**Response:**

```json
{"task_id": "task-004"}
```

Wait for task completion, then retrieve the `secret_id`.

**Step 2: Create LB client certificate secret (optional, for LB to authenticate to backends)**

```
POST /v1/secrets/{project_id}/{region_id}
Content-Type: application/json
```

```json
{
  "name": "lb-client-cert",
  "secret_type": "certificate",
  "payload": "<base64-encoded-lb-client-cert>",
  "payload_content_type": "application/octet-stream",
  "payload_content_encoding": "base64"
}
```

**Response:**

```json
{"task_id": "task-005"}
```

**Step 3: Create Load Balancer with backend mTLS**

```
POST /v1/loadbalancers/{project_id}/{region_id}
Content-Type: application/json
```

```json
{
  "name": "backend-mtls-lb",
  "flavor": "lb1-1-2",
  "vip_network_id": "<network-id>",
  "listeners": [
    {
      "name": "https-listener",
      "protocol": "TERMINATED_HTTPS",
      "protocol_port": 443,
      "insert_x_forwarded": true,
      "pools": [
        {
          "name": "secure-backend-pool",
          "protocol": "HTTPS",
          "lb_algorithm": "ROUND_ROBIN",
          "ca_secret_id": "<backend-ca-cert-id>",
          "secret_id": "<lb-client-cert-id>",
          "members": [
            {
              "address": "192.168.1.20",
              "protocol_port": 443,
              "weight": 1
            }
          ],
          "healthmonitor": {
            "type": "HTTPS",
            "delay": 10,
            "timeout": 5,
            "max_retries": 3,
            "url_path": "/health"
          }
        }
      ]
    }
  ]
}
```

**Traffic flow:** Client → HTTPS → Load Balancer → mTLS → Backend Servers (with certificates)

### Example 3: Adding mTLS to existing Load Balancer

If you need to add mTLS configuration to an existing load balancer and listener:

**Step 1: Create secrets**

Create the required secrets as shown in the previous examples.

**Step 2: Update existing listener**

```
PUT /v1/lblisteners/{project_id}/{region_id}/{listener_id}
Content-Type: application/json
```

```json
{
  "secret_id": "<client-cert-id>"
}
```

**Step 3: Update existing pool**

```
PUT /v1/lbpools/{project_id}/{region_id}/{pool_id}
Content-Type: application/json
```

```json
{
  "protocol": "HTTPS",
  "ca_secret_id": "<backend-ca-cert-id>",
  "secret_id": "<lb-client-cert-id>"
}
```

## Important notes

- **Secret creation timing**: Secrets can be created either before or after the Load Balancer and Listener. If created afterwards, update the Listener/Pool to reference the secrets.

- **Task completion**: Each API call returns a `task_id`. Wait for task completion before proceeding to the next step.

- **Payload format**: Always use `payload_content_type: "application/octet-stream"` when using `payload_content_encoding: "base64"`.

- **Certificate format**:
  - Listener server certificate: PKCS12 format containing certificate, private key, and intermediate chain
  - CA certificates: PEM format, base64 encoded
  - Client certificates: PKCS12 or PEM format, base64 encoded

- **Protocol requirements**:
  - For backend mTLS, pool protocol must be `HTTPS` or `TERMINATED_HTTPS`
  - Health monitor type should match pool protocol (`HTTPS` for secure backends)
