Deploy and run your containerized application in the Cloud, without managing any underlying infrastructure.

## Step 1. Add a container image

<Tip>
**Tip**

If you don't have sufficient resources to a create a Container, request [quota increase](/cloud/getting-started/request-a-quota-increase).
</Tip>

1\. In the Gcore Customer Portal, navigate to **Cloud** > **Container as a Service**.

2\. Click **Create container**.


<Frame>![CaaS page in the Customer Portal](/images/docs/cloud/caas/caas-page.png)</Frame>



3\. In the **Container image** section, select the image type: **public** or **private**. The difference between them is that a private image is secured with credentials.

4\. Enter your image URL. For example, nginx:latest. For a private image, enter the URL in the format `https://registry.mysite.com`.

5\. (Optional) Specify registry credentials. If you selected private image in the previous step, enter credentials for accessing that image. If you've already added credentials to the Customer Portal, choose them from the Credentials dropdown.


<Frame>![Container image settings](/images/docs/cloud/caas/add-a-container/add-container-image.png)</Frame>



If you have no saved credentials, add them as follows:

  * **Image registry name** : Registry name that will be displayed in the Credentials dropdown.
  * **Image registry URL** : Link to the location where your application is stored.
  * **Image registry username** : Username you use to access the storage location of your application.
  * **Image registry password** : Password you use to access the storage location of your application.



To save the new credentials, click **Create credentials**.

(Optional) Set startup command. Enable the toggle if you want to execute a specific command when your container is initiated.

## Step 2. Specify port

In the **Port** section, specify the port for connection to your container.


<Frame>![Container port](/images/docs/cloud/caas/add-a-container/specify-port.png)</Frame>

 <Info>
**Info**

Currently, we support only one port for the container. If you need to add additional data, create a separate container.
</Info>

## Step 3. Choose container configuration

Select the required MB of memory and mCPU (up to 2260 mCPU and 4096 MB). This configuration will be used for the deployed Kubernetes pod, where your container will be placed after creation.


<Frame>![Container configuration](/images/docs/cloud/caas/add-a-container/container-configuration.png)</Frame>



## Step 4. Configure autoscaling

In the **Limits of autoscaling section**, enter the range for the number of nodes you want to maintain under Minimum pods and Maximum pods.


<Frame>![Container autoscaling limits](/images/docs/cloud/caas/add-a-container/autoscaling-limits.png)</Frame>



In the **Cooldown period** field, set the interval (in seconds) between the trigger executions. This helps to prevent frequent and unnecessary scaling changes. You can enter a value between 1 and 3600 seconds.

To ensure more efficient use of computational resources and consistent model performance, define scaling thresholds for CPU, RAM, and HTTP requests resource utilization. You can combine any triggers or use a single one.

In the **Autoscaling triggers**, click **Add trigger** to view and modify current thresholds:

  * The minimum setting is 1% of the resource capacity. Only **HTTP requests** trigger can scale pods to and from 0.

  * The maximum setting is 100% of the resource capacity.



<Frame>![Container autoscaling thresholds](/images/docs/cloud/caas/add-a-container/autoscaling-thresholds.png)</Frame>



By default, the autoscaling parameters are set to 80% but you can enter any percentage within the specified range.

Note that the waiting times specified for the **Cooldown period** and **HTTP requests** trigger are combined, and they determine the total time the system waits before initiating another scaling action.

<Warning>
**Warning**

When setting up the cooldown period or HTTP scaling, use small values with caution because they may affect the scale window.

For the cooldown period, small values can lead to random unexpected scale triggers. For HTTP scaling, small values might affect the time it takes to aggregate HTTP request rates for making scaling decisions.
</Warning>

## Step 5. Specify container lifetime

Enter the number of seconds after which a pod will be deleted when there are no requests to your pod. For example, if you enter 600, the pod will be deleted in 600 seconds - equal to ten minutes.


<Frame>![Container lifetime settings](/images/docs/cloud/caas/add-a-container/container-lifetime.png)</Frame>

 <Tip>
**Tip**

If you specify 0, the container will take approximately one minute to scale down.
</Tip>

## Step 6 (optional). Add environment variables

(Optional) If you want to add metadata to your container, create variables in a form of key-value pairs. These variables will only be used in the environment of the created container.


<Frame>![Enviroment variables settings](/images/docs/cloud/caas/add-a-container/variables.png)</Frame>



For example, if your container supports it, you can configure where an application should write its log files within the container by setting an environment variable for the log file path:

  * Key: `LOG_FILE_PATH`
  * Value: `/var/log/myapp.log`



This variable directs the application to write logs to the specified path inside the container.

## Step 7 (optional). Configure authentication via API

To protect your container endpoints from unauthorized access, enable the API Key authentication feature. Either select an existing API key or create a new one.


<Frame>![API keys settings](/images/docs/cloud/caas/add-a-container/api-key.png)</Frame>



## Step 8. Finalize container configuration

Specify the container name (this will be displayed in the Customer Portal) and additional information if needed.


<Frame>![Container details settings](/images/docs/cloud/caas/add-a-container/container-details.png)</Frame>



In the top-right corner of the screen, click **Create container**. After a few minutes, the initial setup will be finished and the container will be launched.

  <MethodSection id="api" label="REST API">

Container as a Service (CaaS) runs Docker images on a managed Kubernetes backend and exposes each deployment at a public HTTPS endpoint, without cluster administration. Five API calls and three Docker commands provision a private Container Registry, push an image, and start the container.

<Info>
A permanent [API token](/developer-tools/rest-api/authentication) is required, along with a project ID and region ID. Retrieve them from the [Gcore Customer Portal](https://portal.gcore.com) or by calling <code>GET&nbsp;/cloud/v1/projects</code> and <code>GET&nbsp;/cloud/v1/regions</code>. Container Registry and CaaS must be in a region that supports both services - Luxembourg-2 supports both.
</Info>

Open a bash terminal and set these as environment variables before running the examples:

```bash
export GCORE_API_KEY="{YOUR_API_KEY}"
export PROJECT_ID="{YOUR_PROJECT_ID}"
export REGION_ID="{YOUR_REGION_ID}"
```

## Step 1. Create a Container Registry

A Container Registry stores the Docker image before CaaS pulls it. The registry is regional - create it in the same region as the CaaS container.

| Parameter | Required | Description |
|---|---|---|
| `name` | Yes | Registry name. Lowercase letters, numbers, and hyphens. Used as part of the registry URL |
| `storage_limit` | No | Maximum storage in GiB. Defaults to 10 if omitted |

<Tabs>
  <Tab title="curl">
    ```bash
    curl -s -X POST "https://api.gcore.com/cloud/v1/registries/$PROJECT_ID/$REGION_ID" \
      -H "Authorization: APIKey $GCORE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{
        "name": "my-registry",
        "storage_limit": 10
      }'
    ```
  </Tab>
  <Tab title="Python SDK">
    ```python
    import os
    from gcore import Gcore

    client = Gcore(api_key=os.environ["GCORE_API_KEY"])
    registry = client.cloud.registries.create(
        project_id=int(os.environ["PROJECT_ID"]),
        region_id=int(os.environ["REGION_ID"]),
        name="my-registry",
        storage_limit=10,
    )
    print(registry.id, registry.url)
    ```
  </Tab>
</Tabs>

The API returns:

```json
{
  "id": 606,
  "name": "my-registry",
  "storage_limit": 10,
  "storage_used": 0,
  "url": "registry.luxembourg-2.cloud.gcore.dev/1000503-1186668-76-my-registry/"
}
```

```bash
export REGISTRY_ID="{REGISTRY_ID}"
export REGISTRY_URL="registry.luxembourg-2.cloud.gcore.dev/1000503-1186668-76-my-registry"
```

## Step 2. Create a registry user

A registry user provides the credentials for `docker login`. The `secret` field is returned only once at creation time - copy it before proceeding.

| Parameter | Required | Description |
|---|---|---|
| `name` | Yes | Username suffix. Lowercase alphanumeric, max 16 characters. The full Docker login username will be `r_{client}-{project}-{region}-{registry}+{name}` |
| `duration` | Yes | Credential lifetime in days. Use `-1` for no expiration |

<Tabs>
  <Tab title="curl">
    ```bash
    curl -s -X POST "https://api.gcore.com/cloud/v1/registries/$PROJECT_ID/$REGION_ID/$REGISTRY_ID/users" \
      -H "Authorization: APIKey $GCORE_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{
        "name": "pusher",
        "duration": -1
      }'
    ```
  </Tab>
  <Tab title="Python SDK">
    ```python
    registry_user = client.cloud.registries.users.create(
        registry_id=int(os.environ["REGISTRY_ID"]),
        project_id=int(os.environ["PROJECT_ID"]),
        region_id=int(os.environ["REGION_ID"]),
        name="pusher",
        duration=-1,
    )
    print(registry_user.name)    # full Docker login username
    print(registry_user.secret)  # copy now - not retrievable later
    ```
  </Tab>
</Tabs>

The API returns:

```json
{
  "id": 727,
  "name": "r_1000503-1186668-76-my-registry+pusher",
  "read_only": false,
  "duration": -1,
  "secret": "gVa7zL5meW9cBOJLGiQx9AcvPFSehcgg"
}
```

```bash
export DOCKER_USERNAME="r_1000503-1186668-76-my-registry+pusher"
export DOCKER_PASSWORD="{secret_from_response}"
```

<Warning>
The `secret` field is not stored and cannot be retrieved after this response. Save it immediately. If lost, delete the user and create a new one.
</Warning>

## Step 3. Push an image

Log in to the registry, tag a local image with the registry path, and push it. These commands require [Docker](https://docs.docker.com/get-started/get-docker/) installed locally.

```bash
# Log in to the registry
echo "$DOCKER_PASSWORD" | docker login "$REGISTRY_URL" \
  --username "$DOCKER_USERNAME" \
  --password-stdin
```

```
Login Succeeded
```

```bash
# Tag the image with the full registry path
docker tag nginx:latest "$REGISTRY_URL/nginx:latest"

# Push to the registry
docker push "$REGISTRY_URL/nginx:latest"
```

```

```

## Step 4. Create a pull secret

A pull secret stores the registry credentials in the CaaS region, allowing CaaS to pull the private image during container startup. The secret name is referenced by name when creating the container.

| Parameter | Required | Description |
|---|---|---|
| `name` | Yes | Secret name, pattern `^[a-z][-a-z0-9]{0,24}[a-z0-9]$`. Used as reference in the container spec |
| `registry` | Yes | Registry hostname without the path (for example, `registry.luxembourg-2.cloud.gcore.dev`) |
| `login` | Yes | Full Docker username from Step 2 |
| `password` | Yes | Secret from Step 2 |

```bash
curl -s -X POST "https://api.gcore.com/cloud/v1/caas/secrets/$PROJECT_ID/$REGION_ID" \
  -H "Authorization: APIKey $GCORE_API_KEY" \
  -H "Content-Type: application/json" \
  -d "{
    \"name\": \"my-registry-secret\",
    \"registry\": \"registry.luxembourg-2.cloud.gcore.dev\",
    \"login\": \"$DOCKER_USERNAME\",
    \"password\": \"$DOCKER_PASSWORD\"
  }"
```

The API returns:

```json
{
  "name": "my-registry-secret",
  "registry": "registry.luxembourg-2.cloud.gcore.dev",
  "login": "r_1000503-1186668-76-my-registry+pusher",
  "updated_at": "2026-05-27T09:51:00Z"
}
```

```bash
export PULL_SECRET_NAME="my-registry-secret"
```

## Step 5. Create a container

CaaS provisions a pod, pulls the image from the private registry using the pull secret, and exposes the container at a public HTTPS endpoint.

| Parameter | Required | Description |
|---|---|---|
| `name` | Yes | Container name. Pattern `^[a-zA-Z0-9][a-zA-Z0-9.-]+[a-zA-Z0-9]$`, max 43 characters. Included in the public endpoint URL |
| `image` | Yes | Full image path including registry hostname: `{REGISTRY_URL}/nginx:latest` |
| `listening_port` | Yes | Port the container listens on. Must match the application's exposed port |
| `flavor` | Yes | CPU and memory allocation. Run <code>GET&nbsp;/cloud/v1/caas/flavors/{project_id}/{region_id}</code> to list available options |
| `scale.min` | Yes | Minimum number of running pods |
| `scale.max` | Yes | Maximum number of pods for autoscaling |
| `pull_secret` | Yes | Pull secret name from Step 4. Required for private registry images |

```bash
curl -s -X POST "https://api.gcore.com/cloud/v1/caas/$PROJECT_ID/$REGION_ID/containers" \
  -H "Authorization: APIKey $GCORE_API_KEY" \
  -H "Content-Type: application/json" \
  -d "{
    \"name\": \"my-nginx\",
    \"image\": \"$REGISTRY_URL/nginx:latest\",
    \"listening_port\": 80,
    \"flavor\": \"80mCPU-128MiB\",
    \"scale\": {\"min\": 1, \"max\": 2},
    \"pull_secret\": \"$PULL_SECRET_NAME\"
  }"
```

The API returns:

```json
{
  "tasks": ["bf1bbc44-3a20-4c90-bbb9-3781e208fd16"]
}
```

```bash
export TASK_ID="{TASK_ID}"
```

Poll the task every 5 seconds until `state` is `FINISHED`:

```bash
curl -s "https://api.gcore.com/cloud/v1/tasks/$TASK_ID" \
  -H "Authorization: APIKey $GCORE_API_KEY"
```

```json
{
  "id": "bf1bbc44-3a20-4c90-bbb9-3781e208fd16",
  "state": "FINISHED",
  "created_resources": {
    "caas_containers": ["my-nginx"]
  }
}
```

Retrieve the container to get the public endpoint URL:

```bash
CONTAINER_NAME="my-nginx"
curl -s "https://api.gcore.com/cloud/v1/caas/$PROJECT_ID/$REGION_ID/containers/$CONTAINER_NAME" \
  -H "Authorization: APIKey $GCORE_API_KEY"
```

The API returns:

```json
{
  "name": "my-nginx",
  "image": "registry.luxembourg-2.cloud.gcore.dev/1000503-1186668-76-my-registry/nginx:latest",
  "listening_port": 80,
  "flavor": "80mCPU-128MiB",
  "status": "Ready",
  "address": "https://my-nginx-76-1186668-caas.caas.k1.luxembourg-2.cloud.gcore.dev/",
  "deploy_status": {"total": 2, "ready": 1},
  "scale": {"min": 1, "max": 2, "cooldown_period": 60}
}
```

## Step 6. Verify the endpoint

Send a request to the address from the container response. The endpoint is public and requires no authentication by default.

```bash
curl -s -o /dev/null -w "%{http_code}" \
  "https://my-nginx-76-1186668-caas.caas.k1.luxembourg-2.cloud.gcore.dev/"
```

```
200
```

The container is running and serving traffic.
