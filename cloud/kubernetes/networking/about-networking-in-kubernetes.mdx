To effectively manage network traffic within a cluster, understand how Services, Ingress, and Ingress Controllers work together.

### Services

A Service in Kubernetes is an abstraction used to expose an application running on a set of pods. A Service provides access to those pods through a single IP address. Pods can be restarted or recreated and change their internal IP addresses, but they remain accessible through the same Service IP address. A Service distributes incoming traffic across all pods associated with it.

### Ingress

Ingress is a Kubernetes object that defines rules to route external traffic to services inside a cluster. For example, the `/login` route can be mapped to a specific service in the cluster. When a user requests `example.com/login`, traffic is redirected to the service responsible for user logins.

This routing combines three components:

  1. Pods with containers running the application.
  2. A Service that routes traffic to pods.
  3. Ingress that routes traffic from external networks to services.

Ingress defines the routing rules, but these rules have no effect until an Ingress Controller is deployed to enforce them.

### Ingress controllers

An Ingress Controller is a Kubernetes tool that reads ingress objects, builds a routing map of services in the cluster, and directs incoming traffic to the matching service.
