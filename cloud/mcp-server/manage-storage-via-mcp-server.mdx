Gcore MCP Server provides storage management tools that AI clients can invoke through natural language requests. The server returns structured data that the AI client interprets and presents in a readable format.

<Info>
Before starting, [install and configure Gcore MCP Server](/account-settings/integrations/install-and-configure-gcore-mcp-server) with `GCORE_TOOLS="storage"` to load storage tools.
</Info>

## How it works

When a request is made in natural language, the AI client resolves project and region context, calls the appropriate MCP tool, and formats the JSON response for readability.

Example request:

```
List my volumes in Luxembourg
```

The MCP server returns structured data for each volume:

```json
{
  "id": "5ee7968f-fc88-4352-91e0-4869b0b23a46",
  "name": "mcp-test-volume",
  "size": 1,
  "volume_type": "standard",
  "status": "available",
  "bootable": false,
  "is_root_volume": false,
  "limiter_stats": {
    "iops_base_limit": 10,
    "iops_burst_limit": 100,
    "MBps_base_limit": 10,
    "MBps_burst_limit": 100
  },
  "attachments": [],
  "project_id": 1186668,
  "region": "Luxembourg-2",
  "region_id": 76,
  "created_at": "2026-04-20T08:10:37+0000"
}
```

The AI client presents this as readable text:

```
Found 1 volume in Luxembourg-2:

1. mcp-test-volume (1 GB, standard)
   Status: available
   Attached to: none
   IOPS: 10 base / 100 burst
   Throughput: 10 MB/s base / 100 MB/s burst
   Boot volume: no
   Created: 2026-04-20
```

## Volumes

Volumes provide block storage for Virtual Machines. Each VM requires a boot volume for the operating system, and additional data volumes can be attached for application storage.

### List volumes

Listing volumes shows all block storage in a project and region with their type, size, attachment status, and performance characteristics.

```
List my volumes
```

To see only unattached volumes (useful for identifying unused storage):

```
Show unattached volumes
```

Response:

```
Found 2 unattached volumes in Luxembourg-2:

1. old-app-boot (20 GB, ssd_hiiops)
   Created: 2026-01-15
   Bootable: yes (Ubuntu 22.04)
   
2. backup-storage (500 GB, standard)
   Created: 2026-02-15
   Bootable: no

Total unattached storage: 520 GB
```

### View volume details

```
Show details for volume web-server-boot
```

Response includes IOPS limits, attachment info, and image metadata for boot volumes:

```
Volume: web-server-boot
ID: 6632cc34-ac83-4927-8e68-17d1f661ae3d
Size: 50 GB
Type: ssd_hiiops

Performance limits:
  IOPS: 3,000 base / 6,000 burst (2x for 10 minutes)
  Throughput: 100 MB/s base / 200 MB/s burst

Attachment:
  Instance: web-server-01
  Device: /dev/sda
  Attached: 2026-03-01

Boot volume: yes
  OS: Ubuntu 22.04 (linux)
  Architecture: x86_64

Snapshots: 1 available
```

### Volume types

Different volume types suit different workloads. The AI client shows available types when creating volumes.

```
What volume types are available in Luxembourg?
```

Response:

```
Volume types in Luxembourg-2:

1. high_iops_ssd
   IOPS: 60 per GB (max 9,000)
   Throughput: 2.5 MB/s per GB (max 500 MB/s)
   Use case: databases, latency-sensitive workloads

2. standard
   IOPS: 6 per GB (max 4,500)
   Throughput: 0.4 MB/s per GB (max 300 MB/s)
   Use case: general purpose, predictable I/O

3. ssd_lowlatency
   IOPS: up to 5,000
   Latency: ~300 µs average
   Use case: real-time data processing
   Note: snapshots and resizing not supported
```


### Attach volumes to instances

Attaching makes a volume available to the instance as a block device.

```
Attach volume database-storage to instance db-server
```

Response:

```
Attaching volume database-storage to db-server...

Attached successfully:
  Volume: database-storage
  Instance: db-server
  Device: /dev/vdb
  
The volume appears at /dev/vdb in the instance.
Next steps in the instance:
  1. Format: mkfs.ext4 /dev/vdb
  2. Mount: mount /dev/vdb /data
  3. Add to /etc/fstab for persistent mount
```

<Warning>
After attaching, the volume must be formatted and mounted within the instance operating system. The MCP server handles the attachment; filesystem configuration requires connecting to the instance.
</Warning>

### Detach volumes

Detaching disconnects a volume from an instance while preserving data.

```
Detach volume database-storage from instance db-server
```

Response:

```
Detaching volume database-storage...

Detached successfully.
Volume status: available

The volume data is preserved and can be attached to another instance.
```

<Warning>
Unmount the volume within the instance (`umount /data`) before detaching to prevent data corruption.
</Warning>

### Resize volumes

Volumes can be expanded to accommodate growing data needs.

```
Resize volume database-storage to 200 GB
```

Response:

```
Resizing volume database-storage...

Resize completed:
  Previous size: 100 GB
  New size: 200 GB
  
Performance updated:
  IOPS: 9,000 base / 9,000 burst (max reached)
  Throughput: 500 MB/s base / 500 MB/s burst (max reached)

Next: Extend the filesystem in the instance to use the additional space.
  Linux: resize2fs /dev/vdb
  (or: xfs_growfs /data for XFS)
```

<Info>
Volumes can only be increased in size, not decreased. SSD Low-Latency volumes do not support resizing. IOPS and throughput scale with size up to the type maximum.
</Info>

### Delete volumes

```
Delete volume old-backup
```

Response:

```
Deleting volume old-backup...

Volume deleted.
Released storage: 50 GB
```

<Warning>
Attached volumes cannot be deleted. The AI client returns an error listing which instance the volume must be detached from first. Deleted volumes cannot be recovered.
</Warning>

## Snapshots

Snapshots capture the state of a volume at a specific point in time, useful for backups, disaster recovery, and creating volume templates.

### List snapshots

```
List my snapshots
```

Response:

```

  Size: 100 GB
  Source volume: postgres-data (ssd_hiiops)
  Status: available
  Created: 2026-03-19
  Metadata:
    - bootable: false
    - app: postgresql
```


### Create a snapshot

```
Create a snapshot of volume postgres-data named daily-backup
```

Response:

```
Creating snapshot daily-backup...

Snapshot created:
  ID: f7a8b9c0-d1e2-f3a4-b5c6-d7e8f9a0b1c2
  Name: daily-backup
  Source: postgres-data
  Size: 100 GB
  Status: creating

Snapshot will be available in 1-2 minutes.
```

<Info>
For consistent snapshots of databases, stop writes or use application-level quiesce before creating the snapshot. The snapshot captures the volume state at creation time.
</Info>

### Create a volume from snapshot

Restore data or create a copy from a snapshot:

```
Create a volume from snapshot daily-backup named restored-db
```

Response:

```

  Size: 10 GB
  Type: standard (inherited from snapshot)
  Status: available

The volume contains all data from the snapshot point in time.
```


## Troubleshooting

### Volume attachment fails

Common causes and diagnostics:

```
Show details for volume database-storage
```

Check the response for:
- **Status not "available"**: Volume may be attached elsewhere or in error state
- **Region mismatch**: Volume and instance must be in the same region

```
Show details for instance db-server
```

Verify instance is running and not at maximum volume attachment limit.

### Snapshot creation slow

Large volumes with high I/O activity take longer to snapshot.

```
Show status of snapshot daily-backup
```

If stuck in "creating" for more than 30 minutes:
- High write load on source volume can delay snapshot
- Check volume status for errors

### File share not accessible

```
Show details for file share shared-data
```

Verify:
- Status is "available"
- Instance is in the same network as the file share

```
Show network interfaces for instance web-server
```

Check if instance has interface in the file share's network.

### Quota exceeded

```
Show my storage quota
```

Response:

```
Storage quota for project default (Luxembourg-2):

Volumes:
  Used: 12 of 50 volumes
  Size: 320 GB of 1000 GB

Snapshots:
  Used: 1 of 50 snapshots
  Size: 10 GB of 500 GB

File shares:
  Used: 3 of 10 shares
  Size: 210 GB of 500 GB
```

Delete unused volumes, snapshots, or file shares to free quota.

Volume management is documented in [Create and configure volumes](/cloud/virtual-instances/volumes/create-and-configure-volumes). File share setup is covered in [About file shares](/cloud/file-shares/about-file-shares).
