Every request to the Gcore API requires a permanent API token sent in the `Authorization` header. The same token and header format work across all Gcore products — Cloud, CDN, DNS, Storage, and others.

## Create a permanent API token

Tokens are created in the [Gcore Customer Portal](https://portal.gcore.com) under **Profile → API tokens**.

<Info>
The token value is shown only once at creation time. Copy it immediately and store it securely — it cannot be retrieved later.
</Info>

When creating a token, configure two settings:

| Setting | Recommendation |
|---------|---------------|
| **Role** | Use `Administrators` for full API access. Use `Users` to restrict to read-only and limited write operations. A token cannot have a higher role than the user who creates it. |
| **Expiration** | Set an expiration date for tokens used in CI/CD pipelines or by third parties. Use "Never expire" only for long-lived automation that is actively monitored. |

The account limit is 50 tokens. Tokens nearing expiry trigger email notifications at 7 days and 1 day before the expiry date.

The full token management guide is in [API tokens](/account-settings/api-tokens).

## Send the token in requests

Add the token to every request as the `Authorization` header with the `APIKey` scheme:

```bash
curl "https://api.gcore.com/iam/clients/me" \
  -H "Authorization: APIKey {YOUR_TOKEN}"
```

The header name is case-insensitive — `Authorization`, `authorization`, and `AUTHORIZATION` all work. The `APIKey` scheme prefix is required; `Bearer` does not work and returns a token validation error.

### Set the token as an environment variable

Never hardcode a token in a script or source file. Set it as an environment variable instead:

```bash
export GCORE_API_KEY="{YOUR_TOKEN}"

curl "https://api.gcore.com/iam/clients/me" \
  -H "Authorization: APIKey $GCORE_API_KEY"
```

<Warning>
Gcore API tokens always contain a dollar sign (`$`). When the token is used directly inside a double-quoted string in bash, the shell treats everything after `$` as a variable name and silently truncates the token.

**Broken** — shell expands `$c767...` as a variable:
```bash
# Token: 29841$c767b250...
curl ... -H "Authorization: APIKey 29841$c767b250..."
# Sent as: Authorization: APIKey 29841  ← truncated at $
```

**Fixed** — use an environment variable (recommended):
```bash
export GCORE_API_KEY="29841\$c767b250..."  # escape $ in the export
curl ... -H "Authorization: APIKey $GCORE_API_KEY"
```

Or escape the dollar sign directly: `29841\$c767b250...`
</Warning>

## Verify authentication

Call `/iam/clients/me` to confirm the token is valid and see which account it belongs to:

<Tabs>
  <Tab title="curl">
    ```bash
    curl "https://api.gcore.com/iam/clients/me" \
      -H "Authorization: APIKey $GCORE_API_KEY"
    ```
  </Tab>
  <Tab title="Python SDK">
    ```python
    import os
    from gcore import Gcore

    client = Gcore(api_key=os.environ["GCORE_API_KEY"])
    me = client.iam.clients.me()
    print(me.id, me.email)
    ```
  </Tab>
</Tabs>

Response:

```json
{
  "id": 1000503,
  "email": "user@example.com",
  "status": "active",
  "currentUser": 1023101,
  "users": [
    {
      "id": 1023101,
      "name": "Sergey Kostichev",
      "groups": [{"id": 1, "name": "Administrators"}]
    }
  ]
}
```

`id` is the account (client) ID. `currentUser` is the user ID within the account. Both are needed for some IAM API calls.

## Authentication errors

| HTTP status | Message | Cause |
|-------------|---------|-------|
| `401` | `Authentication credentials were not provided.` | No `Authorization` header present |
| `401` | `Bad permanent token: {partial_token}` | Token truncated at `$` — shell variable expansion issue |
| `401` | `Token is invalid or expired` | Token was deleted or has expired |
| `401` | `Given token not valid for any token type` | Using `Bearer` scheme instead of `APIKey` |
| `403` | `You do not have permission to perform this action.` | Token role is insufficient for the requested operation |

## Token scope and roles

A token inherits the role assigned at creation:

- **Administrators** — full read and write access to all resources in the account
- **Users** — read access to most resources; write access restricted to their own resources
- **Engineers** — technical operations without billing and user management access

The token cannot perform actions that the creating user's own role does not permit.

## SSO accounts

When a user authenticates via SAML SSO, the identity provider controls access to the portal — but permanent API tokens operate independently. Revoking SSO access does not invalidate existing tokens. To block API access for an SSO user, delete their tokens manually in the [API tokens](/account-settings/api-tokens) section.
