API protection tags flag and categorize API requests based on conditions and parameters, enabling more efficient traffic management.

These tags are reserved and can't be customized, but they can be used to create custom rules for managing traffic to specific endpoints.

The following reserved tags apply to API protection:

- API Privileged Access
- API Admin Access
- Indicate API Privileged User
- Indicate API Admin User
- Ignore Email Address Detection
- Ignore Phone Number Detection
- Auth Endpoint
- Ignore CCN Detection
- Ignore SSN Detection

Add these tags to API endpoints via the [API Discovery](/waap/api-discovery-and-protection/api-discovery) feature or [custom rules](/waap/waap-rules/custom-rules).

## Configure API protection

API endpoint protection requires three coordinated actions: tagging endpoints by access level, defining matching user role rules, and enabling the [API Protection](/waap/api-protection) policies.

<Info>
For WAAP to correctly identify and protect API endpoints, either enable [API Discovery](/waap/api-discovery-and-protection/api-discovery) or manually add the endpoints to an [API base path](/waap/api-discovery-and-protection/configure-api-base-path).
</Info>

### Step 1. Group endpoints based on their access levels

Categorize API endpoints by authorization level:

- **Admin**: Endpoints accessible only to admin users.
- **Privileged**: Endpoints accessible to privileged users and admins.
- **Non-privileged**: Endpoints accessible to users without elevated permissions.

Custom rules automatically mark endpoints with access tags such as **API Admin Access** or **API Privileged Access**.

When creating a rule, specify secure conditions such as client IP or a specific HTTP response header. Avoid conditions that are easy to spoof, such as the User-agent header.

For example, restrict access to admin endpoints so only users with a valid bearer token in the authorization header can interact with admin APIs.

To create the rule:

1. In the [Gcore Customer Portal](https://portal.gcore.com), navigate to **WAAP** > **Custom Rules**.

2. Click **Add custom rule**.

3. Give the rule a name.

4. In the **IF** section, select **Header**.

5. In the **Header key** field, enter `Authorization`. This means that an admin user has to be authenticated to access the endpoint.

6. In the next dropdown, select **-** to apply the rule only to the specified condition.

7. Select **Contains** as the match criteria to apply the rule even when a request header partly matches the specified one.

8. Enter `Bearer` as the header value to require an authorization token on each request that attempts to access admin API endpoints.

9. In the **Then** section, select the **Tag** action to apply a relevant tag to each request that matches the specified criteria.

10. In the next dropdown, select the relevant reserved tag:

   * **API Privileged Access**: This tag grants permission to reach any privileged API endpoint.

   * **API Admin Access**: This tag allows interaction with all API endpoints. In the current rule example, select this tag.

<Frame>![Custom rule configured to set up privileged API access](/images/docs/waap/api-discovery-and-protection/api-access-with-reserved-tags/privileged-access-APIs-2.png)</Frame>

11. Click **Save** to create the rule.

All requests from authorized users with relevant permissions can now reach the admin API endpoints.

### Step 2. Define user roles that can access endpoints

After defining access levels for endpoints, create a custom rule for each user type to specify which users can interact with those endpoints.

For example, create a rule that identifies admin requests by checking for a valid bearer token in the authorization header.

<Tip>
Adding multiple conditions to custom rules makes them harder to bypass and reduces false positives.
</Tip>

To create the rule:

1. On the **WAAP Rules** page, click **Add custom rule**.

2. Give the rule a name.

3. In the **IF** section, select **Header**.

4. In the **Header key** field, enter `Authorization`. This means that the user has to be authenticated to access the endpoint.

5. In the next dropdown, select **-** to apply the rule only to the specified condition.

6. Select **Contains** as the match criteria to apply the rule even when a request header partly matches the specified one.

7. Enter `Bearer` as the header value to require an authorization token on each request that attempts to access admin API endpoints.

8. Select another condition in the **AND** section:

   * Select **Header** as the condition type.
   * In the **Header key** field, enter `X-Admin-User`.
   * Select **Contains** as the match criteria.
   * Enter `Token` as the header value.

9. In the **Then** section, select the **Tag** action.

10. In the next dropdown, select the relevant reserved tag:

   * **Indicate API Admin User**: Requests from these users will reach any API endpoint. In the current rule example, select this tag.
   * **Indicate API Privileged User**: Requests from these users will reach privileged endpoints.

<Frame>![Custom rule configured to identify privileged users](/images/docs/waap/api-discovery-and-protection/api-access-with-reserved-tags/privileged-users-2.png)</Frame>

11. Click **Save** to create the rule.

All authorized requests that contain the `X-Admin-User` token can now reach the admin API endpoints.

### Step 3 (optional). Configure additional API protection settings

The following optional settings provide additional API protection.

#### Auth token protection

Tag the following OAuth endpoints to protect authentication flows: `oauth/token`, `oauth2/token`, and `oauth2/v1/token`. Protecting these endpoints secures APIs against unauthorized access, token theft, and brute force attacks.

A heuristic mechanism automatically detects and protects Auth token endpoints. For additional customization, manually tag Auth endpoints via custom rules.

This enables more rigorous monitoring and protection for events such as multiple failed login attempts, requests with unauthorized tokens, and repeated requests to forbidden paths.

To tag the endpoints, create the following custom rule:

1. Navigate to **WAAP** > **Custom Rules** and click **Add custom rule**.

2. Give the rule a name.

3. In the **IF** section, select **URL**.

4. In the next dropdown, select **-** to apply the rule only to the specified condition.

5. Select **Equals** as the match criteria to apply the rule when the URL exactly matches the specified one.

6. Enter the relevant endpoint.

7. In the **Then** section, select the **Tag** action to apply a relevant tag to each request that matches the specified criteria.

8. Select the **Auth Endpoint** reserved tag.

<Frame>![Custom rule configured to protect an Auth endpoint](/images/docs/waap/api-discovery-and-protection/api-access-with-reserved-tags/auth-endpoint-2.png)</Frame>

9. Click **Save** to create the rule.

#### Ignore sensitive data exposure detection

Use tags to notify WAAP of legitimate PII exposure in request responses — such as social security numbers, credit card numbers, email addresses, and phone numbers — or to exclude requests from the [data exposure](/waap/waap-policies/waf-and-owasp-top-threats) policy, which covers broader exposures such as confidential organizational data and code leaks.

This lets legitimate resources collect user information without triggering WAAP interruptions while keeping sensitive data leakage protection active. Ignore rules also reduce false positives, preventing WAAP from blocking requests to specific endpoints.

For instance, create the following rule to ignore data collection during the checkout process:

1. Navigate to **WAAP** > **Custom Rules** and click **Add custom rule**.

2. Give the rule a name.

3. In the **IF** section, select **URL**.

4. In the next dropdown, select **-** to apply the rule only to the specified condition.

5. Select **Contains** as the match criteria to apply the rule when the URL partially matches the specified one.

6. Enter the relevant endpoint. In this example, `/url/order/checkout`.

7. In the **Then** section, select the **Tag** action to apply a relevant tag to each request that matches the specified criteria.

8. Select the **Ignore Phone Number Detection** and **Ignore Email Address Detection** tags.

<Frame>![Custom rule configured to ignore sensitive data detection](/images/docs/waap/api-discovery-and-protection/api-access-with-reserved-tags/ignore-sde-detection-2.png)</Frame>

9. Click **Save** to create the rule.

### Step 4. Enable API protection policies

After adding access tags to restricted API endpoints and configuring user roles, enable the relevant [API Protection](/waap/api-protection) policies.

<Frame>![API Protection page showing the available protection policies](/images/docs/waap/waap-policies/advanced-api-protection/advanced-api-protection-2.png)</Frame>

With these policies enabled, only users with admin and privileged roles can interact with those endpoints after logging in.

<Info>
Admin users can access any endpoint, so define which users hold admin roles before enabling the policies.
</Info>

After the policy is enabled, WAAP blocks any request that doesn't carry the correct user tag.
