Gcore MCP Server provides Web Application and API Protection (WAAP) management through natural language. AI assistants can protect domains, investigate attacks, create security rules, and monitor threats.

<Info>
Before starting, [install and configure Gcore MCP Server](/account-settings/integrations/install-and-configure-gcore-mcp-server) with `GCORE_TOOLS="waap.*"` to load WAAP tools.
</Info>

## How it works

When a request is made in natural language, the AI client calls the appropriate MCP tool and presents the response in readable format.

Example request:

```
List my protected domains
```

The AI client responds:

```
Found 2 protected domains:

1. shop.example.com
   Status: monitoring (logging threats, not blocking)
   Custom rules: 2 of 5 used
   Advanced rules: 1 of 3 used
   Created: October 21, 2025

2. api.example.com
   Status: blocking (active protection)
   Custom rules: 3 of 5 used
   Advanced rules: 2 of 3 used
   Created: September 15, 2025
   
Recommendation: shop.example.com has been in monitoring mode 
for 2 months. Review blocked events and consider switching to 
blocking mode if no false positives detected.
```

## Protect a new domain

When onboarding a new domain, the AI assistant guides through the complete setup.

```
I need to protect my e-commerce site shop.example.com with WAAP
```

Response:

```
Setting up WAAP protection for shop.example.com...

Domain registered:
  ID: 1456
  Name: shop.example.com
  Status: monitoring (default - safe for initial setup)
  
DNS Configuration Required:
  Add this CNAME record at your DNS provider:
  
  shop.example.com  CNAME  shop-example-com.waap.gcore.com
  
  Or if using Gcore DNS: "Add CNAME for shop.example.com 
  pointing to shop-example-com.waap.gcore.com"

After DNS propagation (typically 5-30 minutes), traffic will 
route through WAAP for inspection.

Default protection enabled:
  - WAF policies (OWASP Top 10)
  - Protocol validation
  - IP reputation filtering
  - Known bot detection

Next steps:
1. Wait for DNS propagation
2. Monitor traffic for 24-48 hours
3. Review events for false positives
4. Switch to blocking mode when confident

Would you like me to configure additional protection like 
rate limiting or geo-blocking?
```

## Create security rules

### Rate limiting

```
Create a rate limiting rule: max 60 requests per minute per IP for api.example.com
```

Response:

```
Creating rate limiting rule...

Rule created:
  ID: rule_78901
  Domain: api.example.com
  Type: Rate Limit
  
Configuration:
  Threshold: 60 requests per minute
  Scope: per IP address
  Action: block (returns 429 Too Many Requests)
  
The rule is active immediately. IPs exceeding 60 requests/minute 
will be blocked for the remainder of that minute.

Current rate limit rules (1 of 3 allowed):
  - rule_78901: 60 req/min per IP (all endpoints)

To create endpoint-specific limits:
"Add rate limit of 10 requests per minute for /api/login endpoint"
```

### Geo-blocking

```
Block all traffic from Russia and China for shop.example.com
```

Response:

```
Creating geo-blocking rule...

Rule created:
  ID: rule_78902
  Domain: shop.example.com
  Type: Geo-Block
  
Configuration:
  Blocked countries: Russia (RU), China (CN)
  Action: block (returns 403 Forbidden)
  
Visitors from these countries will see the block page.

Impact estimate (based on last 30 days):
  Traffic from RU: 2.3% of total (mostly bot traffic)
  Traffic from CN: 1.8% of total (mixed)
  Blocked attacks from these regions: 67%

If you have legitimate customers in these regions, consider:
- Allowlisting specific IPs
- Using challenge mode instead of block

Modify: "Change geo-block rule to challenge mode instead of block"
```

## Troubleshooting

### Legitimate traffic being blocked

```
Users are reporting they can't access shop.example.com from their office
```

Response:

```
Let me investigate...

Checking recent blocks for shop.example.com...

Found potential issue:
  IP 203.0.113.100 blocked 45 times in last hour
  Reason: Rate limit exceeded (rule_78901)
  
This appears to be a corporate network (single IP for 
multiple users behind NAT).

Options:
1. Allowlist this IP
   "Add IP 203.0.113.100 to allowlist for shop.example.com"
   
2. Increase rate limit for this IP
   "Create exception: 500 req/min for IP 203.0.113.100"
   
3. Adjust global rate limit
   "Increase rate limit to 120 req/min per IP"

Recommendation: Option 1 (allowlist) if this is a known 
corporate customer. This bypasses rate limiting while 
maintaining other protections.

Which option would you like?
```

### False positives in WAF

```
Our checkout form is being blocked by WAF
```

Response:

```
Investigating WAF blocks on checkout...

Found issue:
  Path: /checkout/payment
  Blocked: 234 requests today
  Rule triggered: SQL Injection detection
  
False positive identified:
  The payment form field "card_notes" contains user text 
  that matches SQL injection patterns (e.g., users typing 
  "don't charge until...")

Solutions:

1. Create WAF exception for this endpoint
   "Disable SQL injection check for /checkout/payment path"
   Risk: Low (payment data validated server-side)
   
2. Exclude specific parameter
   "Exclude parameter 'card_notes' from WAF inspection"
   Risk: Very low (only this field excluded)

3. Adjust WAF sensitivity
   "Set WAF to low sensitivity for shop.example.com"
   Risk: Medium (may miss some attacks)

Recommendation: Option 2 - exclude only the problematic 
parameter while maintaining full protection elsewhere.

Implement? "Exclude parameter 'card_notes' from WAF for 
shop.example.com"
```
