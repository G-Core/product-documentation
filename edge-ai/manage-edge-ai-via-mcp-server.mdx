Gcore MCP Server provides Edge AI management through natural language. AI assistants can provision GPU infrastructure for model training and deploy inference endpoints - the same operations available in the Customer Portal or API.

<Info>
Before starting, [install and configure Gcore MCP Server](/account-settings/integrations/install-and-configure-gcore-mcp-server) with `GCORE_TOOLS="ai,ai_ml,gpu_baremetal,gpu_virtual"` to load Edge AI tools.
</Info>

## When to use Edge AI

Edge AI covers two stages of the machine learning workflow:

**Training** - GPU clusters for training models from scratch, fine-tuning existing models, or running computationally intensive experiments. Requires GPU Cloud infrastructure.

**Inference** - Deploying trained models to serve predictions with low latency. Everywhere Inference distributes models across edge locations and routes requests to the nearest endpoint.

## Training infrastructure with GPU Cloud

GPU clusters provide the compute power for training machine learning models. Before creating a cluster, determine the requirements based on your workload.

### Choose the right cluster type

Three cluster types are available, each suited to different use cases:

| Cluster type | Best for | Trade-offs |
|--------------|----------|------------|
| **Bare Metal** | Production training, long-running jobs | Highest performance, fixed configuration |
| **Spot Bare Metal** | Fault-tolerant training, development | Up to 50% cheaper, can be interrupted with 24h notice |
| **Virtual** | Development, variable workloads | Flexible (can resize), slightly lower performance |

To understand what's available before provisioning:

```
What GPU cluster options do I have for training a large language model?
```

The AI assistant explains available configurations:

```
For LLM training, consider these options:

Bare Metal with InfiniBand (recommended for distributed training):
  - H100 cluster: 8x H100 80GB per node, 3.2 Tbit/s InfiniBand
    Best for: Training models that don't fit on a single node
    
  - A100 cluster: 8x A100 80GB per node, 800 Gbit/s InfiniBand
    Best for: Multi-node training with moderate communication needs

Without InfiniBand:
  - A100 Ethernet: 8x A100 80GB per node
    Best for: Single-node training, independent batch jobs
    
  - L40S: 8x L40S 48GB per node
    Best for: Fine-tuning smaller models, inference

InfiniBand is essential for distributed training where gradients sync 
frequently between nodes. For single-node work, Ethernet is sufficient.

What size model are you training? I can recommend the right configuration.
```

```

```

<Info>
GPU cluster provisioning involves physical server allocation. Bare Metal clusters take 15-40 minutes; Virtual clusters are faster at 5-10 minutes.
</Info>

## Deploy models with Everywhere Inference

After training (or using a pre-trained model), deploy it for production inference. Everywhere Inference distributes your model across edge locations and routes requests to the nearest endpoint automatically.

### Understand inference deployment

```
I have a trained model. How does Everywhere Inference work?
```

Response:

```
Everywhere Inference deploys your model to edge locations worldwide:

1. You provide a model (from our catalog or your own container)
2. We deploy replicas across regions you choose
3. Users get a single endpoint (https://your-model.inference.gcore.com)
4. Requests route automatically to the nearest healthy replica

Benefits:
  - Low latency: requests go to geographically closest GPU
  - High availability: automatic failover if a region has issues
  - Auto-scaling: replicas scale based on traffic

Pricing: Pay per GPU-hour while replicas are running
```

```
My inference costs are too high. What can I do?
```

Response:

```
Analyzing deployment llama3-70b-xxxx for cost optimization...

Current cost: ~$X/hour (3 replicas, 2x L40S each)

Options:

1. Right-size GPU allocation
   Your model uses 140GB VRAM. Currently using 2x L40S (96GB).
   Alternative: 2x A100 80GB would give 160GB with better performance.
   Impact: Similar cost but faster inference.

2. Reduce replica count during off-peak
   Traffic drops 70% between 02:00-08:00 UTC.
   Scale to 1 replica during off-peak, back to 3 for peak.
   Estimated savings: 25-30%

3. Use smaller model for simple queries
   If some requests are simple, route them to a 7B model.
   Keep 70B for complex queries only.
   Estimated savings: 40-50% depending on query distribution

4. Enable request batching
   Group concurrent requests to maximize GPU efficiency.
   Works well if latency targets are >500ms.
   Estimated savings: 10-20%

Which approach would you like to explore?
```
