Managed Kubernetes in GPU Cloud provides a fully managed Kubernetes environment optimized for GPU workloads. Clusters support standard virtual instances, bare metal servers, and GPU bare metal nodes with NVIDIA H100 GPUs. GPU bare metal availability and specific GPU models vary by region.

Unlike traditional Kubernetes deployments that require manual control plane management, Managed Kubernetes handles the control plane automatically. The focus shifts to deploying and scaling AI/ML workloads rather than cluster administration.

## Cluster architecture

Each Kubernetes cluster consists of:

- **Control plane**: Managed by Gcore, handles scheduling, API server, and cluster state
- **Worker node pools**: Groups of nodes with identical configuration where workloads run

Node pools can contain different instance types within the same cluster. For example, a cluster might have one pool of standard virtual machines for general workloads and another pool of GPU bare metal nodes for training jobs.

All nodes in a pool share the same configuration: instance type, volume settings, and network configuration. The cluster autoscaler can add or remove nodes within each pool based on demand.

## Prerequisites

Before creating a cluster, ensure the following resources are configured:

| Resource | Description |
|----------|-------------|
| Network | A VxLAN or VLAN network in the target region |
| Subnet | A subnet within the network with available IP addresses |

Networks and subnets can be created during cluster creation or configured beforehand in **GPU Cloud** > **Networking** > **Networks**.

## Create a Kubernetes cluster

To create a Managed Kubernetes cluster in GPU Cloud:

1. In the [Gcore Customer Portal](https://portal.gcore.com), navigate to **GPU Cloud**.
2. In the sidebar, select **Managed Kubernetes**.
3. Click **Create Cluster**.

<img src="/images/docs/edge-ai/managed-kubernetes/managed-kubernetes-overview.png" alt="Managed Kubernetes page in GPU Cloud showing clusters list and Create Cluster button" />

### Step 1. Select region

In the **Region** section, select the data center location for the cluster. Regions are grouped by geography:

- **Americas**: Manassas
- **Asia-Pacific**: Incheon
- **EMEA**: Chester, London-2, Luxembourg-2, Luxembourg-3, Sines-2, Sines-3

<img src="/images/docs/edge-ai/managed-kubernetes/step1-region.png" alt="Region selection showing available data centers grouped by Americas, Asia-Pacific, and EMEA" />

GPU bare metal availability and supported features such as VAST Storage integration vary by region.

<Info>
Changing the region after configuring other settings resets all form values.
</Info>

### Step 2. Select Kubernetes version

In the **Kubernetes cluster version** dropdown, select the desired version.

The latest stable version is selected by default. Kubernetes versions follow the standard release cycle. Newer versions include the latest features and security patches. The version can be upgraded after cluster creation through the cluster overview page.

### Step 3. Configure node pools

Node pools define the worker nodes where workloads run. Each pool has its own instance type, scaling settings, and volume configuration.

1. In the **Pools** section, configure the default pool or click **Add pool** to create additional pools.

2. Enter a **Pool name** to identify the pool.

3. Set **Minimum nodes** and **Maximum nodes** for autoscaling. The cluster autoscaler adds nodes when pods are pending due to resource constraints and removes nodes when they are underutilized.

4. Select the node **Type**:

| Type | Description | Use case |
|------|-------------|----------|
| Virtual instances | Standard VMs | Development, testing, general workloads |
| Bare metal instances | Dedicated physical servers without GPU | High-performance CPU workloads |
| GPU Bare metal instances | NVIDIA H100 GPU servers with NVMe storage | Model training, inference at scale |

<Note>
Bare metal instances are not available in all regions. If the option is greyed out, switch to a region that supports bare metal, such as Luxembourg-2.
</Note>

5. Select a **Flavor**. Each flavor card displays specifications, storage, and hourly pricing. For GPU Bare metal instances, the flavor includes built-in NVMe storage—no separate volume configuration is required.

<img src="/images/docs/edge-ai/managed-kubernetes/gpu-flavors-dropdown.png" alt="GPU Bare metal flavor selection showing H100 specifications and pricing" />

6. For Virtual and Bare metal instance types, configure the boot volume:
   - **Volume type**: High IOPS SSD or SSD Low-Latency
   - **Size**: Minimum 50 GiB

7. Select a **Placement policy**. Anti-affinity soft distributes nodes across different physical hosts when possible.

8. Configure optional node settings:
   - **Add labels**: Kubernetes labels for node selection in pod specs
   - **Add taints**: Prevent pods from scheduling on nodes unless they tolerate the taint
   - **Autohealing nodes**: Automatically replace unhealthy nodes (enabled by default)
   - **Public IPv4 address**: Assign public IPs to nodes for direct internet access

9. To configure advanced pool settings such as container runtime capabilities, environment variables, and resource limits, click the **...** menu in the pool header and select **Configure settings**.

<Warning>
Advanced pool settings cannot be changed after cluster creation. GPU bare metal nodes may be deleted and recreated during scaling or maintenance. Use external storage such as Object Storage, NFS, or managed databases for persistent data.
</Warning>

### Step 4. Configure CNI provider

The Container Network Interface (CNI) provider manages pod networking within the cluster.

<img src="/images/docs/edge-ai/managed-kubernetes/step4-cni.png" alt="CNI provider section with Calico and Cilium options, CIDR configuration, and network monitoring settings" />

| Provider | Description |
|----------|-------------|
| Calico | Established CNI with broad compatibility |
| Cilium | Modern eBPF-based CNI with advanced observability (default) |

<Info>
The CNI provider cannot be changed after cluster creation. Choose based on long-term requirements.
</Info>

Configure network ranges within the same section:

- **Pod CIDR**: IP range for pods (default: 172.16.0.0/17)
- **Service CIDR**: IP range for Kubernetes services (default: 172.24.0.0/17)
- **Node mask size**: Subnet size allocated to each node (default: /24)

When Cilium is selected, the **Enable Hubble for network monitoring** option adds network observability and flow visibility. IPv6 dual-stack is available as a Beta feature in supported regions.

### Step 5. Configure network settings

In the **Network settings** section, configure the private network interface for cluster nodes.

<img src="/images/docs/edge-ai/managed-kubernetes/step5-network.png" alt="Network settings showing network and subnetwork selection dropdowns" />

1. Select an existing **Network** or click **Add a new network** to create one.
2. Select a **Subnetwork** within the chosen network.

The subnet determines the IP range for node interfaces. Ensure the subnet has sufficient available IP addresses for the maximum number of nodes across all pools. All cluster nodes receive Basic DDoS Protection at no additional cost.

### Step 6. Name the cluster

In the **Cluster name** field, enter a name or use the auto-generated default.

### Step 7. Configure logging

The **Logging** section enables log collection to OpenSearch Dashboards for debugging and monitoring. Logging is a paid feature.

<Note>
Logging is not available in all regions. In GPU-focused regions such as Sines-3, the section displays "Logging is currently not supported in this region."
</Note>

Toggle **Enable Logging** to enable it when available, then configure the log topic.

### Step 8. VAST Storage integration

In regions where VAST Storage integration is available, the **VAST Storage integration** section displays an **Enable VAST Storage** toggle.

<img src="/images/docs/edge-ai/managed-kubernetes/vast-storage-logging.png" alt="VAST Storage integration section showing the Enable VAST Storage toggle" />

<Note>
VAST Storage integration is available in select GPU regions. Contact Gcore support to confirm availability and enable it for your account before creating a cluster.
</Note>

### Step 9. Configure advanced settings

The **Advanced settings** section contains optional cluster-level configuration:

**OIDC authentication**: Enable Single Sign-On for the Kubernetes API using an external identity provider such as Okta, Azure AD, or Keycloak. Configure the issuer URL, client ID, and claim settings.

**Cluster Autoscaler**: Configure autoscaling behavior including scan intervals, scale-down thresholds, scale-up delays, and node grace periods.

### Step 10. Create the cluster

Review the **Estimated cost** panel on the right to verify monthly and hourly pricing for all configured resources, including 17% VAT.

If your account lacks sufficient quota, the portal displays "You need to request quota increase to create this resource" and the **Create Cluster** button is greyed out. Click **Send quota request** to submit a quota increase request, then create the cluster once the request is approved.

Click **Create Cluster** to begin provisioning.

Cluster provisioning time depends on the number and type of nodes. The cluster status changes to **Provisioned** when ready.

## Connect to the cluster

The kubeconfig file enables kubectl connections to the cluster. Download it after the cluster reaches **Provisioned** status:

1. Navigate to the cluster overview page.
2. Click **Kubernetes config**.
3. Save the downloaded `k8sConfig.yml` file.

Configure kubectl to use the downloaded configuration:

```bash
export KUBECONFIG=/path/to/k8sConfig.yml
```

Verify the connection:

```bash
kubectl cluster-info
kubectl get nodes
```

Nodes display **Ready** status when fully provisioned.

For detailed cluster management including adding GPU node pools, configuring autoscaling parameters, upgrading Kubernetes versions, and viewing audit logs, see [Manage a GPU Kubernetes cluster](/edge-ai/managed-kubernetes/manage-a-gpu-kubernetes-cluster).
